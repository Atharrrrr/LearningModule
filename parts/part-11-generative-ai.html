Autoencoders

Every model so far has learned to map an input to some other useful output — a label, a caption, a next token. An autoencoder does something stranger-sounding at first: it learns to map an input back to itself, through a deliberately narrow bottleneck. That constraint turns out to force something genuinely useful to emerge.

✏️ Frame of reference

An artist studies a complex scene, then jots down a short list of essential notes — "three trees, a red barn, sunset light" — without looking at the scene again. Later, working only from that short list, they sketch the scene from memory. If the sketch is close to the original, the short list genuinely captured what mattered. The shorter that list can be while still allowing a good sketch, the more efficiently the scene's real structure has been captured.

A network trained to reproduce its own input

An autoencoder has two halves: an encoder that compresses an input down into a latent representation — a vector far smaller than the input itself — and a decoder that reconstructs the original input from that small vector alone. Training minimizes reconstruction error (Chapter 4.3): how different the decoder's output is from the original input, using gradient descent same as ever. The narrow bottleneck is the entire point — it's what forces the encoder to discard anything inessential and keep only what the decoder actually needs.

● Interactive demo — try it

A real, working (if simple) autoencoder: encoder(x) = the average of x's 3 numbers (compressing 3 numbers into 1); decoder(z) = [z, z, z] (broadcasting that 1 number back into 3). Real reconstruction error, computed live, for 5 different input patterns.

Every pattern gets compressed to exactly one number and reconstructed from exactly that one number — the same bottleneck size throughout. But the reconstruction error is 100× larger for Pattern 5 than for the other four. That's not a flaw in the autoencoder; it's an honest measurement of how much real structure a 1-number bottleneck can actually capture, for a given input. Patterns 1-4 happened to be internally correlated — all three of their numbers move together — so a single number (their average) was nearly enough. Pattern 5 varies a lot across its three positions, and one number simply isn't enough information to rebuild it.

This is exactly Chapter 4.9's dimensionality reduction lesson, now with a network doing the compressing and reconstructing rather than a fixed linear projection like PCA. A real trained autoencoder uses a learned encoder and decoder — often with nonlinear activation functions (Chapter 6.3) — letting it capture more complex structure than a simple average, but the underlying principle is identical: the bottleneck forces the network to keep only what's needed to reconstruct, and how well it succeeds depends entirely on how much genuine structure the input has to begin with.

✓ What you can now explain to someone else
  • An autoencoder is a network trained to reconstruct its own input, through an encoder-decoder pair with a narrow bottleneck in between.
  • The bottleneck — the latent representation — is forced to be much smaller than the input, so it can only capture the input's essential structure.
  • Reconstruction error measures how much information the bottleneck failed to preserve; it depends both on the bottleneck's size and on how much genuine internal structure the input actually has.
  • Autoencoders generalize Chapter 4.9's dimensionality reduction: a learned, potentially nonlinear compression and reconstruction, instead of a fixed linear projection like PCA.
Quick check
1. What is an autoencoder trained to do?
Reconstruct its own input, after compressing it down through a narrow bottleneck and back out again.
Predict a completely separate label for its input, like a classifier.
Expand its input into a much larger representation than it started with.
2. In the demo, why did Pattern 5 reconstruct so much worse than Patterns 1-4, even though every pattern used the exact same 1-number bottleneck?
Patterns 1-4's three numbers were close to each other, so their average nearly captured all of them; Pattern 5's three numbers genuinely differed, so one number couldn't represent all three well.
Pattern 5 used a larger bottleneck than the other four patterns.
The reconstruction error was computed differently for Pattern 5 than for the others.
3. How does a real trained autoencoder generalize beyond Chapter 4.9's PCA?
It uses a learned encoder and decoder, often with nonlinear activation functions, instead of PCA's fixed linear projection — letting it capture more complex structure.
An autoencoder has nothing in common with PCA's dimensionality reduction idea.
An autoencoder always uses a bigger bottleneck than PCA would use.
