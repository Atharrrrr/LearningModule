Part XI · Chapter 11.1

Autoencoders

Every model so far has learned to map an input to some other useful output — a label, a caption, a next token. An autoencoder does something stranger-sounding at first: it learns to map an input back to itself, through a deliberately narrow bottleneck. That constraint turns out to force something genuinely useful to emerge.

✏️ Frame of reference

An artist studies a complex scene, then jots down a short list of essential notes — "three trees, a red barn, sunset light" — without looking at the scene again. Later, working only from that short list, they sketch the scene from memory. If the sketch is close to the original, the short list genuinely captured what mattered. The shorter that list can be while still allowing a good sketch, the more efficiently the scene's real structure has been captured.

A network trained to reproduce its own input

An autoencoder has two halves: an encoder that compresses an input down into a latent representation — a vector far smaller than the input itself — and a decoder that reconstructs the original input from that small vector alone. Training minimizes reconstruction error (Chapter 4.3): how different the decoder's output is from the original input, using gradient descent same as ever. The narrow bottleneck is the entire point — it's what forces the encoder to discard anything inessential and keep only what the decoder actually needs.

● Interactive demo — try it

A real, working (if simple) autoencoder: encoder(x) = the average of x's 3 numbers (compressing 3 numbers into 1); decoder(z) = [z, z, z] (broadcasting that 1 number back into 3). Real reconstruction error, computed live, for 5 different input patterns.

Every pattern gets compressed to exactly one number and reconstructed from exactly that one number — the same bottleneck size throughout. But the reconstruction error is 100× larger for Pattern 5 than for the other four. That's not a flaw in the autoencoder; it's an honest measurement of how much real structure a 1-number bottleneck can actually capture, for a given input. Patterns 1-4 happened to be internally correlated — all three of their numbers move together — so a single number (their average) was nearly enough. Pattern 5 varies a lot across its three positions, and one number simply isn't enough information to rebuild it.

This is exactly Chapter 4.9's dimensionality reduction lesson, now with a network doing the compressing and reconstructing rather than a fixed linear projection like PCA. A real trained autoencoder uses a learned encoder and decoder — often with nonlinear activation functions (Chapter 6.3) — letting it capture more complex structure than a simple average, but the underlying principle is identical: the bottleneck forces the network to keep only what's needed to reconstruct, and how well it succeeds depends entirely on how much genuine structure the input has to begin with.

✓ What you can now explain to someone else
  • An autoencoder is a network trained to reconstruct its own input, through an encoder-decoder pair with a narrow bottleneck in between.
  • The bottleneck — the latent representation — is forced to be much smaller than the input, so it can only capture the input's essential structure.
  • Reconstruction error measures how much information the bottleneck failed to preserve; it depends both on the bottleneck's size and on how much genuine internal structure the input actually has.
  • Autoencoders generalize Chapter 4.9's dimensionality reduction: a learned, potentially nonlinear compression and reconstruction, instead of a fixed linear projection like PCA.

Part XI · Chapter 11.2

Variational Autoencoders

Chapter 11.1's autoencoder compresses each input down to one exact point in latent space. Nothing guarantees that a nearby point — one the encoder never actually produced — decodes into anything sensible at all; the space could have "holes" between real data points that decode into garbage. A variational autoencoder fixes exactly this, and in doing so turns an autoencoder into something that can generate brand-new outputs.

✏️ Frame of reference

Chapter 11.1's artist wrote down one exact note-list for a scene: "three trees, a red barn, sunset light." A variational artist writes something looser instead: "sunset-ish — could lean a bit more orange, could lean a bit more pink." That's not one point, it's a small, described range of plausible notes. Hand that range to a second artist who's never seen the scene, and whichever specific note-list they happen to pick from within it, they can still sketch something plausible — because the range itself was built to always describe something reasonable, not just one exact memorized case.

Encoding to a distribution instead of a single point

A variational autoencoder (VAE) changes what the encoder outputs: instead of a single latent number, it outputs a mean and a standard deviation — the two parameters of a normal distribution (Chapter 2.6). To reconstruct an input, the decoder doesn't receive that mean directly; it receives a random sample drawn from the distribution those two numbers describe (Chapter 2.5's probability, put to direct use). Training pushes two things at once: that sampled reconstruction should still resemble the original input, and — via an extra regularization pressure (Chapter 5.4) — every input's distribution should stay close to a shared, well-behaved shape, so the latent space doesn't fragment into isolated, disconnected regions.

● Interactive demo — try it

Encoding Pattern 1 [0.90, 0.80, 0.85] the VAE way: mean = 0.85 (same as Chapter 11.1's average), standard deviation = 0.05. Each click draws one real Gaussian sample (Box-Muller transform, verified statistically) from that distribution and decodes it — a different, but always plausible, reconstruction every time.

Click "Sample" several times: every z is a genuinely different random number, so every reconstruction is slightly different too — yet every single one stays close to the original pattern, because they're all drawn from a distribution centered right on it. Chapter 11.1's plain autoencoder only ever produced exactly one reconstruction for exactly one input; this one produces a whole family of plausible variations, all real, all different, none identical.

That's the entire payoff: once a VAE's latent space is trained to keep every region well-behaved this way, you don't need a real input to get a plausible output at all — pick literally any point in the latent space, sample around it, decode it, and get something new the model never saw during training. That's genuine generation, not just compression and reconstruction, and it's the foundation the rest of this Part builds on.

✓ What you can now explain to someone else
  • A plain autoencoder encodes each input to one exact latent point, with no guarantee that nearby points decode to anything sensible.
  • A VAE's encoder outputs a distribution (mean and standard deviation) instead, and the decoder reconstructs from a random sample of that distribution.
  • Training pushes both good reconstruction from sampled points and a regularization pressure keeping the whole latent space well-organized, not fragmented.
  • Because sampling anywhere reasonable in a VAE's latent space decodes to something plausible, a VAE can generate genuinely new outputs — not just reconstruct inputs it already saw.
