Part XI · Chapter 11.1

Autoencoders

Every model so far has learned to map an input to some other useful output — a label, a caption, a next token. An autoencoder does something stranger-sounding at first: it learns to map an input back to itself, through a deliberately narrow bottleneck. That constraint turns out to force something genuinely useful to emerge.

✏️ Frame of reference

An artist studies a complex scene, then jots down a short list of essential notes — "three trees, a red barn, sunset light" — without looking at the scene again. Later, working only from that short list, they sketch the scene from memory. If the sketch is close to the original, the short list genuinely captured what mattered. The shorter that list can be while still allowing a good sketch, the more efficiently the scene's real structure has been captured.

A network trained to reproduce its own input

An autoencoder has two halves: an encoder that compresses an input down into a latent representation — a vector far smaller than the input itself — and a decoder that reconstructs the original input from that small vector alone. Training minimizes reconstruction error (Chapter 4.3): how different the decoder's output is from the original input, using gradient descent same as ever. The narrow bottleneck is the entire point — it's what forces the encoder to discard anything inessential and keep only what the decoder actually needs.

● Interactive demo — try it

A real, working (if simple) autoencoder: encoder(x) = the average of x's 3 numbers (compressing 3 numbers into 1); decoder(z) = [z, z, z] (broadcasting that 1 number back into 3). Real reconstruction error, computed live, for 5 different input patterns.

Every pattern gets compressed to exactly one number and reconstructed from exactly that one number — the same bottleneck size throughout. But the reconstruction error is 100× larger for Pattern 5 than for the other four. That's not a flaw in the autoencoder; it's an honest measurement of how much real structure a 1-number bottleneck can actually capture, for a given input. Patterns 1-4 happened to be internally correlated — all three of their numbers move together — so a single number (their average) was nearly enough. Pattern 5 varies a lot across its three positions, and one number simply isn't enough information to rebuild it.

This is exactly Chapter 4.9's dimensionality reduction lesson, now with a network doing the compressing and reconstructing rather than a fixed linear projection like PCA. A real trained autoencoder uses a learned encoder and decoder — often with nonlinear activation functions (Chapter 6.3) — letting it capture more complex structure than a simple average, but the underlying principle is identical: the bottleneck forces the network to keep only what's needed to reconstruct, and how well it succeeds depends entirely on how much genuine structure the input has to begin with.

✓ What you can now explain to someone else
  • An autoencoder is a network trained to reconstruct its own input, through an encoder-decoder pair with a narrow bottleneck in between.
  • The bottleneck — the latent representation — is forced to be much smaller than the input, so it can only capture the input's essential structure.
  • Reconstruction error measures how much information the bottleneck failed to preserve; it depends both on the bottleneck's size and on how much genuine internal structure the input actually has.
  • Autoencoders generalize Chapter 4.9's dimensionality reduction: a learned, potentially nonlinear compression and reconstruction, instead of a fixed linear projection like PCA.

Part XI · Chapter 11.2

Variational Autoencoders

Chapter 11.1's autoencoder compresses each input down to one exact point in latent space. Nothing guarantees that a nearby point — one the encoder never actually produced — decodes into anything sensible at all; the space could have "holes" between real data points that decode into garbage. A variational autoencoder fixes exactly this, and in doing so turns an autoencoder into something that can generate brand-new outputs.

✏️ Frame of reference

Chapter 11.1's artist wrote down one exact note-list for a scene: "three trees, a red barn, sunset light." A variational artist writes something looser instead: "sunset-ish — could lean a bit more orange, could lean a bit more pink." That's not one point, it's a small, described range of plausible notes. Hand that range to a second artist who's never seen the scene, and whichever specific note-list they happen to pick from within it, they can still sketch something plausible — because the range itself was built to always describe something reasonable, not just one exact memorized case.

Encoding to a distribution instead of a single point

A variational autoencoder (VAE) changes what the encoder outputs: instead of a single latent number, it outputs a mean and a standard deviation — the two parameters of a normal distribution (Chapter 2.6). To reconstruct an input, the decoder doesn't receive that mean directly; it receives a random sample drawn from the distribution those two numbers describe (Chapter 2.5's probability, put to direct use). Training pushes two things at once: that sampled reconstruction should still resemble the original input, and — via an extra regularization pressure (Chapter 5.4) — every input's distribution should stay close to a shared, well-behaved shape, so the latent space doesn't fragment into isolated, disconnected regions.

● Interactive demo — try it

Encoding Pattern 1 [0.90, 0.80, 0.85] the VAE way: mean = 0.85 (same as Chapter 11.1's average), standard deviation = 0.05. Each click draws one real Gaussian sample (Box-Muller transform, verified statistically) from that distribution and decodes it — a different, but always plausible, reconstruction every time.

Click "Sample" several times: every z is a genuinely different random number, so every reconstruction is slightly different too — yet every single one stays close to the original pattern, because they're all drawn from a distribution centered right on it. Chapter 11.1's plain autoencoder only ever produced exactly one reconstruction for exactly one input; this one produces a whole family of plausible variations, all real, all different, none identical.

That's the entire payoff: once a VAE's latent space is trained to keep every region well-behaved this way, you don't need a real input to get a plausible output at all — pick literally any point in the latent space, sample around it, decode it, and get something new the model never saw during training. That's genuine generation, not just compression and reconstruction, and it's the foundation the rest of this Part builds on.

  • Because sampling anywhere reasonable in a VAE's latent space decodes to something plausible, a VAE can generate genuinely new outputs — not just reconstruct inputs it already saw.

Part XI · Chapter 11.3

GANs

Chapter 11.2's VAE generates by sampling a well-organized latent space and decoding. A GAN takes a completely different route to the same goal — realistic generated output — by pitting two networks against each other and letting competition, not a reconstruction target, do the training.

🕵️ Frame of reference

An art forger and an authenticator are locked in an ongoing contest. The forger paints fakes, trying to fool the authenticator; the authenticator studies real and fake paintings side by side, trying to get better at telling them apart. Every time the authenticator improves, the forger has to improve too, or every fake gets caught. Every time the forger improves, the authenticator has to sharpen their eye further. Neither one ever "finishes" studying — they improve because of each other, round after round.

Two networks, trained against each other

A generative adversarial network (GAN) trains a generator and a discriminator at the same time, with opposing goals. The discriminator is trained as an ordinary classifier (Chapter 4.2, Chapter 6.1's neuron): shown a mix of real examples (labeled "real") and the generator's current output (labeled "fake"), it adjusts its weights via gradient descent to tell them apart better. The generator never sees real data directly at all — it's trained only to adjust its own output so the discriminator scores it higher as "real" next time. Each one's improvement changes the target the other is chasing, which is exactly what makes this "adversarial."

● Interactive demo — try it

A real, working 1-number toy GAN. Real data is fixed at 0.8. The generator starts by outputting 0.1; the discriminator starts barely able to tell anything apart. Each click runs one real training round: a discriminator gradient-descent step, then a generator gradient-ascent step against that updated discriminator — the exact same math as a real GAN, just with one number standing in for a whole image.

Click through 30-40 rounds: the generator's output climbs from 0.1 toward the neighborhood of the real value 0.8, while D(real) and D(fake) drift toward each other, both settling near 0.5 — the discriminator genuinely can't tell real from fake apart anymore, which is exactly the sign a GAN is supposed to reach. A real image-generating GAN runs the identical idea at far larger scale: the generator takes random noise and produces a full image through a decoder-like network, the discriminator is a real CNN classifier (Chapter 7.2), and both train for many thousands of rounds instead of 40 — but the core loop, one network's gradient chasing the other's current weakness, is exactly what just ran above.

  • Adversarial training genuinely oscillates rather than converging smoothly, since each network's target keeps shifting as the other one improves; a well-trained GAN reaches a point where the discriminator can barely distinguish real from fake at all.

Diffusion Models

Chapter 11.3's GAN generates an entire realistic output in one single, difficult forward pass — and pays for it with the unstable, oscillating training seen in that chapter's demo. Diffusion models take the opposite trade: instead of one hard generation step, they use hundreds or thousands of very small, very easy steps. That trade turns out to be remarkably effective, and today's leading image generators are built on it.

📺 Frame of reference

Imagine a photograph slowly dissolving into television static — not all at once, but one tiny sprinkle of static added at a time, over a thousand tiny steps, until nothing recognizable remains. Now imagine training a restorer whose only job, ever, is this: given an image with one sprinkle too much static, guess what it looked like one sprinkle earlier. That's a much easier job than "turn pure static into a finished photo" in one leap. But run that tiny, easy correction a thousand times in a row, starting from pure static, and a coherent photograph can gradually emerge — built entirely out of small, individually modest corrections.

Forward diffusion: gradually destroying structure, on purpose

The forward diffusion process takes real data and destroys it on purpose, a little at a time: at each step, mix in a small amount of random Gaussian noise (Chapter 2.6, the same real sampling mechanism as Chapter 11.2's VAE demo). After enough steps, whatever structure the original data had is completely gone, and the result looks exactly like random noise. Nothing here is learned — it's a fixed recipe, and because it's fixed and mechanical, exactly how much noise was added at every single step is always known.

Reverse diffusion: undoing it, one small step at a time

Reverse diffusion (denoising) is where learning happens. During training, the network is shown a noisy sample from some step of the forward process and asked to predict the specific noise that was mixed in to produce it — a genuine, known target, since the forward process is fixed and fully recorded. That's an ordinary supervised regression problem (Chapter 4.3), repeated across every step of the noise schedule. Once trained, generation reverses the direction entirely: start from pure random noise, run the trained denoiser to estimate and remove a small amount of it, repeat that hundreds or thousands of times, and a coherent, novel sample gradually emerges — never needing to jump straight from noise to a finished result in one leap.

● Interactive demo — try it

A real 1-number forward/reverse diffusion process. Forward step: x ← √(1−β)·x + √β·noise, with noise a real Gaussian sample (Chapter 11.2's Box-Muller transform) and β=0.05. Starting value x₀ = 0.8 — click forward until it's unrecognizable as noise, then click reverse to undo it exactly, step by step, using the real noise values that were actually added.

Click "Forward" 15-20 times: the value wanders far from 0.8 and becomes statistically indistinguishable from any other random number. This demo can cheat — it recorded the real noise added at every step, so reversing it is exact algebra, not prediction. A real trained diffusion model doesn't get to see the real noise; it only has an approximate, learned guess at each step. But run that approximate guess enough times, in the same small-step-by-small-step structure just demonstrated here, and it reliably converges to a coherent result anyway.

Why trade one hard step for a thousand easy ones?

Chapter 11.3's GAN generator has to be right about an entire realistic output in a single forward pass, trained against a constantly shifting discriminator — genuinely unstable, as that chapter's wobbling demo showed directly. A diffusion model's network only ever has to solve one much narrower problem: "given this noise level, what noise was just added?" — the same modest regression question, answered the same way, at every step. That narrower, repeated task is a large part of why diffusion models train more stably than GANs, at a real cost: generating one sample means running the network many times in sequence, not once.

✓ What you can now explain to someone else
  • The forward diffusion process is a fixed, unlearned procedure that gradually adds Gaussian noise to real data until it becomes indistinguishable from pure noise.
  • The reverse diffusion process is a trained network that predicts the noise added at each step, since the forward process's real noise is always a known training target.
  • Generation runs the trained denoiser backward from pure random noise, removing a small amount at each step, until a coherent new sample emerges.
  • Diffusion trades a GAN's single hard, unstable generation step for many small, easy denoising steps — more stable to train, at the cost of needing many sequential steps to generate one sample.

Part XI · Chapter 11.5

Text-to-Image (Diffusion Meets CLIP)

Chapter 11.4's diffusion model can generate a realistic-looking image from pure noise — but nothing about that process, on its own, lets you ask for a specific image. Text-to-image models add exactly one more ingredient: at every single denoising step, steer the process using Chapter 10.1's CLIP-style text understanding, so the image that emerges matches a written prompt, not just "looks realistic in general."

🗿 Frame of reference

A sculptor works a rough block of marble down toward a finished statue, one small chisel strike at a time — Chapter 11.4's gradual denoising. Now imagine a critic standing beside them the entire time, holding a written description, and murmuring after every single strike: "more like that, less like this." Each individual strike is still just a small, modest correction — but with that steady whisper guiding every one of them, the statue that gradually emerges resembles the written description specifically, not just "some statue or other."

Conditioning every denoising step on a text prompt

Chapter 11.4's denoiser, unconditioned, only ever answers "what noise was likely added here?" — nothing points it toward any particular subject. Text-to-image diffusion adds conditioning: encode the text prompt using a CLIP-style text encoder (Chapter 10.1), and feed that embedding into the denoising network at every single step, through the same attention mechanism (Chapter 8.4) that's already inside the network — the image being denoised supplies the queries, and the text embedding supplies the keys and values, letting every part of the image "look at" the prompt at every step. The denoiser's prediction shifts accordingly: not just "plausible noise to remove," but "noise to remove in a way that steers toward this prompt."

● Interactive demo — try it

A real trained model doesn't move directly toward a fixed target this way — the text embedding shifts its noise prediction instead — but the outcome shown here is real and honest: identical starting noise, different prompt, different result.

Nothing about the starting noise changed between the two runs; only the conditioning did. That's the entire mechanism, scaled up from one number to millions of pixels: the same trained denoiser, the same step-by-step process, steered toward wildly different results by whatever text prompt's embedding is feeding in at every step.

This is why text-to-image models can produce an enormous range of outputs from one trained network: the diffusion process (Chapter 11.4) supplies the "generate something realistic" half, and CLIP's shared embedding space (Chapter 10.1) supplies the "make it match this description" half — two ideas from two different chapters, combined into one guided generation loop.
