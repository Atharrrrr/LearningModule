Meet the VLM (CLIP)

Chapter 7 gave a model the ability to understand images. Chapters 8-9 gave a model the ability to understand and generate text. A vision-language model connects the two: it learns a single shared space where both a picture and a sentence describing it can be represented as the same kind of thing.

🗂️ Frame of reference

Imagine a librarian filing both photographs and written descriptions into the same set of drawers, organized purely by subject matter — every photo of a dog, and every caption that describes a dog, lands in the same drawer, regardless of whether it's a picture or a sentence. Ask for "a photo of a dog" and the librarian can pull the exact right photo from that drawer, having never needed the two to look alike in any literal sense — only to belong to the same drawer. That shared filing system, for images and text at once, is exactly what a vision-language model learns.

One shared embedding space for two different kinds of input

A VLM (vision-language model) uses two separate encoders — one for images, built from Chapter 7's CNN or Transformer-based vision architectures, and one for text, built from Chapter 8-9's Transformer machinery — but trains them together so that both encoders land their outputs in the exact same embedding space. An image of a dog and the sentence "a photo of a dog" end up as two nearby points in that one shared space, exactly like Chapter 8.3's word embeddings, but now spanning two different modalities at once.

CLIP: learning the space by contrasting matches against mismatches

CLIP is the model that made this approach practical at scale. It trains on hundreds of millions of real (image, caption) pairs collected from the web — no one had to hand-label what's "in" each image; the existing caption is the label. Contrastive learning is the mechanism: take a batch of image-caption pairs, compute every image's embedding and every caption's embedding, and use gradient descent (Chapter 4.3) to push each image's embedding toward high cosine similarity with its own caption, and toward low similarity with every other caption in that batch. Repeated over enough batches, the space organizes itself so "matching content" means "nearby embeddings" — for images and text alike.

Every image retrieves its true matching caption with similarity above 0.998, while every mismatched pairing scores far lower — even negative for the most unrelated pair (dog photo vs. "a photo of a car," at -0.874). That wide, reliable gap between matches and mismatches is exactly what contrastive training is built to produce, and it's the same nearest-neighbor mechanism Chapter 9.7's RAG used for document retrieval — just applied across images and text instead of within text alone.

This shared space is what makes CLIP so reusable: it powers image search from a text query, zero-shot image classification (comparing an image's embedding against candidate class-name captions with no task-specific training at all), and — as Chapter 11.5 covers later — guiding image generation itself.

✓ What you can now explain to someone else
  • A vision-language model uses separate image and text encoders, trained together so both land in the same shared embedding space.
  • CLIP trains on real (image, caption) pairs from the web, using contrastive learning: pull matched pairs together, push mismatched pairs apart, purely from cosine similarity in that shared space.
  • No manual labeling of image contents is needed — the existing caption that came with each image is the training signal.
  • A well-trained CLIP-style space reliably separates true matches (similarity near 1) from mismatches (similarity low or negative), the same nearest-neighbor mechanism behind Chapter 9.7's RAG retrieval.

Part X · Chapter 10.2

Captioning & Visual Question Answering

Chapter 10.1's shared embedding space is enough to retrieve a matching caption out of a fixed list — but real captioning systems generate a brand-new sentence, and real visual question answering (VQA) systems answer a question that was never written down anywhere in advance. Both build directly on Chapter 9.2's autoregressive generation, just conditioned on an image as well as (or instead of) text.

🖼️ Frame of reference

A museum tour guide standing in front of a painting can do two different things. Unprompted, they might describe it: "a portrait of a woman in a blue dress, seated by a window." Or a visitor might ask something specific: "what color is her dress?" — and the guide answers that exact question instead of giving the general description. Same painting, same understanding of it, two different outputs depending on whether there's a specific question to answer.

Captioning: generate a description, conditioned on an image

Image captioning takes Chapter 9.2's autoregressive generation and swaps out the starting context: instead of conditioning purely on preceding text tokens, the model conditions on an image's features first, then generates a caption one token at a time, same mechanism, same "predict the most likely next token given everything so far" — "everything so far" now includes the image itself.

VQA: answer a specific question about a specific image

Visual question answering (VQA) adds one more piece of context: both the image's features and a specific question's tokens, together, condition the same autoregressive generation. Ask a different question about the exact same image, and the generated answer changes accordingly — the model isn't just describing the image in general, it's answering the particular thing that was asked.

✓ What you can now explain to someone else
  • Image captioning generates a description by conditioning autoregressive generation (Chapter 9.2) on an image's features instead of, or in addition to, preceding text.
  • VQA conditions that same generation on both an image's features and a specific question, producing a targeted answer rather than a general description.
  • Captioning and VQA share the same underlying mechanism — what changes is what's included as context going into generation.
  • Different questions about the same image can produce very different answers, because each one conditions generation on different question tokens layered on top of the same image features.
