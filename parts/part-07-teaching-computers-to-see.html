Part VII · Chapter 7.1

Images Are Just Grids of Numbers

Everything in this Part — recognizing cats, detecting cars, matching keypoints between photos — is built on one simple fact: a computer never actually sees a picture. It sees a grid of numbers, and nothing else.

🏟️ Frame of reference

A stadium card display, where thousands of fans each hold up a single colored card on cue. From high in the stands, it resolves into a giant picture. Walk down onto the field, though, and the picture disappears — all you see is one person holding one card with one specific color. The picture was never actually "in" the display; it only ever existed as many individually addressable colors, arranged in a grid. A digital image works exactly the same way.

The pixel: one number (or three)

A pixel is one cell in that grid. In a grayscale image, each pixel is a single number from 0 (black) to 255 (white), with everything in between a shade of gray. Stack pixels into rows and columns and you get exactly Chapter 2.3's matrix: one row per image row, one column per image column, one number per pixel.

With the numbers hidden, this reads as a picture — a plus sign. Reveal them, and it's obviously just a grid of two numbers, 230 and 60, arranged in a pattern. Nothing about the image "knows" it's a plus sign; that's a pattern a human — or, as the rest of this Part shows, a trained model — recognizes in the numbers.

Color: three grids stacked together

A color image adds two more grids. Each channel — red, green, and blue — is its own grayscale grid, exactly like the plus sign above. Combine the three channels' values at each position, and you get a color. This is precisely Chapter 2.11's tensor: height × width × 3 stacked matrices, one per channel.

  • A computer never "sees" a picture directly; every operation in this Part works on these numbers.

Part VII · Chapter 7.2

Convolutional Neural Networks

Feed a whole image into Chapter 6.2's fully-connected layers, and two problems show up immediately: an enormous number of weights, and a network with no built-in sense that nearby pixels are related. Convolution fixes both, using an idea that's already familiar.

🔍 Frame of reference

Scanning a whole page of text for the letter "e," using a small magnifying glass. You don't need a different expert stationed at every position on the page — the same small "does this look like an e?" check gets slid across every patch of the page, one small window at a time. The exact same detector, reused everywhere, rather than one giant detector custom-built for the entire page at once.

The convolution: a dot product, slid across an image

A kernel (filter) is a small grid of weights — commonly 3×3. At every position on the image, it multiplies its weights against the pixels underneath it and sums the results: exactly Chapter 2.4's dot product, computed between the kernel and whatever small patch of the image it's currently sitting over. Slide it across every position, and each dot product becomes one value in a feature map — a convolutional neural network (CNN) is built from layers of exactly this operation.

The vertical-edge kernel lights up wherever brightness changes left-to-right — the plus sign's left and right edges — and goes completely flat (zero) down the middle and outer columns, where nothing changes horizontally. The horizontal-edge kernel does the mirror opposite, lighting up the plus sign's top and bottom edges. Same operation, different weights, different feature detected — this is the entire idea a CNN's early layers learn automatically, rather than needing a human to hand-pick "detect vertical edges" as a rule.

Why not just use a fully-connected layer?

A fully-connected layer (Chapter 6.2) mapping this 8×8 = 64-pixel image to a 6×6 = 36-value output would need a separate weight for every input-output pair: 64 × 36 = 2,304 weights. The convolution above produced that same 36-value output using just 9 weights — the kernel — reused at all 36 positions. That weight sharing is the whole efficiency story: a pattern worth detecting in one corner of an image is usually just as worth detecting anywhere else, so the same small detector gets reused everywhere instead of learned separately, over and over, per position.

Stack several convolutional layers and something else emerges: early layers detect simple things like edges (exactly what's shown above); layers built on top of those combine edges into corners and textures; layers built on top of those combine textures into parts and shapes. Chapter 7.5 tours the famous architectures built this way.

✓ What you can now explain to someone else
  • A kernel is a small grid of weights that computes a dot product with the image patch underneath it, at every position.
  • Sliding a kernel across an image produces a feature map, showing where that kernel's pattern is strongly present.
  • The same kernel is reused at every position (weight sharing), using far fewer parameters than a fully-connected layer over the same input.
  • Stacked convolutional layers build up from simple features (edges) to increasingly complex ones (textures, parts, shapes).

Part VII · Chapter 7.3

The Classic CV Toolbox (SIFT/ORB)

Long before CNNs (Chapter 7.2) learned features automatically, computer vision found distinctive points in images using hand-engineered rules. Those classic techniques are still used today, and directly set up Chapter 7.8's SuperPoint — a learned replacement for exactly this idea.

🏠 Frame of reference

Recognizing your own house in a stranger's photo album, even though every photo was taken at a different angle, time of day, and zoom level. You don't compare every pixel — you scan for a few unmistakable landmarks (the red door, a distinctive window shape) and check whether those specific landmarks show up. Classic computer vision automates exactly that: find a handful of distinctive points, then describe each one well enough to recognize it again elsewhere.

Keypoints: points worth remembering

A keypoint is a point worth remembering: not just any pixel, but one distinctive enough to find again. Flat regions and straight edges make poor keypoints — a flat patch of sky looks identical wherever you crop it, and a point on a straight edge could have slid along that edge and looked the same. A corner, where intensity changes sharply in more than one direction at once, is exactly the kind of point that's genuinely pinpoint-able.

Descriptors: fingerprints for matching

Finding a keypoint is only half the job — you also need a way to recognize the same keypoint in a different image. A descriptor is a vector (Chapter 2.2) summarizing what the area around a keypoint looks like. Matching two keypoints across images becomes a distance calculation between their descriptor vectors — smaller distance means a better match, exactly the similarity idea from Chapter 2.4.

SIFT vs. ORB

SIFT and ORB are the two most well-known classic detector-and-descriptor pairs. Both find keypoints and build descriptors for them; they differ mainly in speed and descriptor design:

  • SIFT — descriptors are longer vectors of real numbers, very robust to scale and rotation changes, but slower to compute and compare.
  • ORB — descriptors are compact binary strings (matched with fast bitwise comparisons instead of Euclidean distance), dramatically faster, free to use, slightly less robust than SIFT.

Both were, for years, the standard way to stitch panoramas, track objects across video, and reconstruct 3D scenes from photos — all without a single learned parameter. Chapter 7.8 picks this exact thread back up with SuperPoint, which learns both the keypoint detector and the descriptor instead of hand-designing them.

✓ What you can now explain to someone else
  • A keypoint is a distinctive, repeatable point — typically a corner, where intensity changes sharply in more than one direction.
  • A descriptor is a vector summarizing a keypoint's local appearance, used to match it against keypoints in another image via vector distance.
  • SIFT uses accurate but slower real-valued descriptors; ORB uses fast, compact binary descriptors — a speed/accuracy trade-off.
  • These classic, hand-engineered techniques predate CNNs and still directly motivate learned successors like Chapter 7.8's SuperPoint.

Part VII · Chapter 7.4

Classification vs. Detection vs. Segmentation

"What does a vision model output?" doesn't have one answer — it depends entirely on the question being asked. Three different questions about the same photo produce three genuinely different kinds of output.

🖼️ Frame of reference

Three different requests to a photo editor, about the same photo. "Is there a dog in this photo?" needs one word: "yes." "Where is the dog?" needs a box drawn around it. "Cut the dog out perfectly, so I can put it on a different background" needs every single pixel that belongs to the dog traced out, exactly — nothing more, nothing less. Same photo, three completely different amounts of detail.

Three tasks, three output shapes

Classification (Chapter 4.2) answers "what's the main thing here?" with a single label for the whole image — no location information at all.

Object detection answers "what's here, and where?" — every object gets its own bounding box and label. This handles multiple objects naturally, but a box is a loose fit: it includes some background inside its rectangle, and says nothing about the object's exact outline.

Segmentation answers "exactly which pixels?" — a label for every single pixel, tracing each object's true outline instead of a loose rectangle around it. It's the most detailed of the three, and also the most expensive to produce training data for, since every pixel needs a human-verified label rather than one label per image.

✓ What you can now explain to someone else
  • Classification outputs one label for the whole image — no location.
  • Detection outputs a bounding box and label per object — location, but only a loose rectangle, not an exact outline.
  • Segmentation outputs a label for every pixel — the exact outline, at the highest labeling cost.
  • Each task is strictly more detailed (and more expensive to produce training data for) than the one before it.

Part VII · Chapter 7.5

Famous Architectures & Transfer Learning

CNNs (Chapter 7.2) got dramatically deeper and more capable over a roughly 15-year stretch — and one architecture in particular solved a problem this book already met head-on. That same history also explains why almost nobody trains a vision model completely from scratch anymore.

👨‍🍳 Frame of reference

Hiring an experienced chef instead of training a new cook from absolute zero. An experienced chef already has years of general technique — knife skills, heat control, seasoning instinct — that transfers to almost any cuisine. Teaching them your restaurant's specific menu takes an afternoon, not years. Only a total beginner needs to relearn how to hold a knife from scratch.

A short, famous timeline

A few architectures marked real turning points:

  • LeNet (1998) — one of the first real CNNs, reading handwritten digits. Small by today's standards, but proof the idea worked.
  • AlexNet (2012) — a much deeper CNN, trained on GPUs, that dramatically beat every non-deep-learning approach on ImageNet (a huge labeled image dataset). This is widely seen as deep learning's breakout moment for vision.
  • VGG (2014) — showed that stacking many small 3×3 kernels (Chapter 7.2), rather than a few large ones, reliably improved results as depth increased.
  • ResNet (2015) — pushed depth further still (over 100 layers), by directly fixing the vanishing gradient problem from Chapter 6.7.

ResNet's fix: give the gradient a shortcut

A plain layer computes y = f(x). A residual connection (skip connection) instead computes y = f(x) + x — the input skips straight past the layer and gets added back in. That "+x" matters enormously for backpropagation: the gradient of y with respect to x becomes f'(x) + 1 instead of just f'(x). Even if the learned path's gradient is small, adding 1 keeps the total gradient from collapsing toward zero.

● Interactive demo — try it

Chapter 6.7's exact sigmoid-derivative scenario (factor 0.25 per layer), compared plain vs. with residual connections.

At 20 layers, the plain network's gradient has collapsed to about 9×10⁻¹³ — completely vanished, exactly Chapter 6.7's story. The residual version's gradient is still around 87 — nowhere close to vanishing. That one structural change is what let networks go from tens of layers to hundreds.

Transfer learning: reusing what a network already knows

Chapter 7.2 showed early CNN layers learn simple, generic features like edges; later layers combine those into increasingly task-specific patterns. That generic-to-specific progression is exactly what makes transfer learning work: take a model already trained on a huge, general dataset, keep its early layers exactly as they are (they've already learned generically useful features), and only fine-tune the later layers for a new, specific task. The "experienced chef" doesn't relearn knife skills for a new menu.

● Interactive demo — try it

Freeze all 4 early layers, and fine-tuning only touches the final layer — under 1% of the network's total parameters, yet often enough to reach strong performance on a new task with far less data and training time than starting from scratch. Chapter 7.6 covers how to actually measure whether a vision model — from scratch or fine-tuned — is any good.

✓ What you can now explain to someone else
  • Famous CNN architectures (LeNet, AlexNet, VGG, ResNet) trace a path toward greater depth and capability.
  • A residual connection adds a layer's input directly to its output, giving the gradient a "+1" shortcut that keeps it from vanishing (Chapter 6.7) even in very deep networks.
  • Transfer learning reuses a pretrained model's early, general-purpose layers, fine-tuning only the later, task-specific layers on new data.
  • Freezing early layers can reduce the number of parameters that need retraining to a small fraction of the total network.

Part VII · Chapter 7.6

Evaluating Vision Models (IoU/mAP)

Chapter 5.3's precision, recall, and F1 all assumed a prediction was simply right or wrong. A detector's bounding box (Chapter 7.4) is never exactly right or wrong — it's a rectangle that overlaps the true answer by some amount. Vision needs its own way to turn "close" into a number before precision and recall can even apply.

🚗 Frame of reference

Grading how well a car is parked inside a marked space. A car centered perfectly in the lines overlaps the marked space almost completely. A car parked crooked, half over the line, overlaps only partially — plenty of marked space sits empty, and plenty of car sits outside it. The fraction of the combined car-plus-space area that's actually shared by both is a single, honest number for "how well parked is this?"

IoU: turning overlap into a number

IoU (Intersection over Union) is exactly that fraction: overlap area ÷ combined area. A detection is usually only counted as correct if its IoU with the true box clears some threshold — commonly 0.5.

● Interactive demo — try it

A fixed 4×4 ground-truth box (green) and a same-size predicted box (orange) you can shift. Real computed IoU, live.

At zero shift the boxes match exactly — IoU = 1.0. Shift by just one row or column and IoU already drops to 0.6, since a chunk of each box no longer overlaps the other. Push the shift far enough and the boxes stop touching entirely — IoU = 0, no matter how "close" the boxes still look on a busy image.

From IoU to a precision-recall curve

Once IoU decides whether one detection counts as a true positive or a false positive, everything from Chapter 5.3 — precision, recall — applies directly. A detector doesn't output just one box per object; it outputs many candidate boxes, each with a confidence score. Rank them by confidence, and precision and recall trace out a curve as you accept more and more detections.

That single number — precision, averaged at each point a new true positive appears — is a class's Average Precision (AP). Compute AP separately for every class a detector handles, then average those across all classes, and you get mAP (mean Average Precision) — the standard headline number used to compare object detectors, including Chapter 7.7's YOLO.

✓ What you can now explain to someone else
  • IoU measures how well two boxes overlap: intersection area ÷ union area, from 0 to 1.
  • A detection typically only counts as a true positive if its IoU with the ground truth clears a threshold (commonly 0.5).
  • Once IoU decides true/false positives, Chapter 5.3's precision and recall apply directly to ranked detections.
  • Average Precision summarizes a class's precision-recall trade-off in one number; mAP averages AP across all classes.

Part VII · Chapter 7.7

Meet YOLO

Chapter 7.4 defined what detection outputs — boxes and labels. Chapter 7.6 defined how to grade it — IoU and mAP. This chapter meets a real detector that's actually used in production: YOLO, built around one genuinely clever trick for getting boxes out of an image fast.

📸 Frame of reference

Two ways to find every friend's face in a packed party photo. The slow way: circle a bunch of candidate regions first, then go back and examine each circled region individually — two separate passes over the photo. The fast way: glance at the whole photo exactly once, and while looking, simultaneously mark down where every face is and who it belongs to, all in that one glance. YOLO — "You Only Look Once" — is named for doing exactly the second thing.

One grid, one pass, every box at once

YOLO (You Only Look Once) divides the image into an S×S grid. Every cell predicts, all at once, in one forward pass: whether an object's center falls inside it, a bounding box for that object, and which class it is. Older detectors ran two separate stages — propose regions, then classify each — costing multiple passes through the network. YOLO's entire output comes from one pass, which is exactly why it's fast enough for real-time video.

The rule for which cell "owns" an object is simple and exact: whichever grid cell contains that object's center is responsible for predicting it — not every cell the object's box happens to overlap, just the one holding its center point.

Cleaning up: non-max suppression

In practice, nearby cells often all fire somewhat for the same real object, producing several overlapping boxes instead of one. Non-max suppression (NMS) cleans this up: sort every candidate box by confidence, keep the top one, throw out any remaining box whose IoU with it clears a threshold (Chapter 7.6), then repeat with whatever boxes are left.
