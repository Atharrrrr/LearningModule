Part XII · Chapter 12.1

Learning by Trial and Reward

Every model so far has learned from labeled examples (Chapter 1.2's supervised learning) or from patterns in unlabeled data. Reinforcement learning is different from both: nobody ever hands the learner a correct answer. It just tries things, sees what happens, and gradually gets better at getting more of whatever counts as "good."

🐕 Frame of reference

Nobody trains a dog to sit by showing it a labeled dataset of "correct sitting" photos. Instead, the dog tries something, and if it happens to sit, it gets a treat; if it wanders off, it doesn't. Repeat that enough times, across enough attempts, and the dog's behavior shifts toward whatever earns treats more often — not because anyone described the "correct" behavior in words, but purely because trying things and noticing which ones paid off, over and over, was enough.

The trial-and-error setup

Reinforcement learning's basic loop repeats forever: an agent (Chapter 1.1 already introduced this word — Chapter 9.8's LLM agents borrowed their name from this original, more general meaning) observes a state of the environment, picks an action according to its current policy, and receives a reward plus a new state. Nothing in that loop ever tells the agent "the correct action was X" — it only ever finds out how much reward an action it actually took happened to earn. Reinforcement learning is the problem of turning that stream of tried-actions-and-rewards into a policy that earns more reward over time.

Pull each arm 10-15 times: no arm ever tells you its true odds directly, but the running averages — computed from real, genuinely random outcomes — drift toward revealing which arm actually pays out more often (Arm B, with a hidden 70% chance, versus Arm A's 30% and Arm C's 50%). That's the entire mechanism of learning from trial and error: no labels, no correct answers handed over — just accumulating evidence from your own actions' real outcomes, and letting that evidence guide which action you'd favor next.

This tiny setup — pick an action, get a reward, no state or sequence involved at all — is called a "multi-armed bandit" problem, the simplest possible reinforcement learning setting. Real reinforcement learning problems add back everything this demo left out: a changing state the agent must react to, and actions whose consequences unfold over many future steps, not just an immediate reward. Chapter 12.2 introduces the first real algorithm for handling that fuller picture.

✓ What you can now explain to someone else
  • Reinforcement learning has no labeled correct answers (Chapter 1.2's supervised learning) — an agent only ever learns from the rewards its own actions actually earned.
  • The basic loop is: observe a state, pick an action according to a policy, receive a reward and a new state, repeat.
  • A policy is the agent's current strategy for choosing actions; the whole goal of reinforcement learning is improving that policy to earn more total reward over time.
  • Even the simplest setting — a multi-armed bandit, with no state or sequence at all — shows the core idea: real observed rewards, accumulated over repeated trials, reveal which actions are actually better.

Part XII · Chapter 12.2

Q-Learning, Simply Explained

Chapter 12.1's bandit had no state and no delay: pull an arm, get an immediate reward, done. Real problems rarely work that way — where you are matters, and a good action now might not pay off for several more steps. Q-learning is the classic algorithm for learning good behavior in exactly that fuller, more realistic setting.

🚚 Frame of reference

A delivery driver learning a new route doesn't get paid the instant they turn left at some intersection — the payoff for that turn might not arrive until several more turns later, when it finally leads to the depot. Over enough trips, the driver builds up a sense of "how good is turning left from this specific intersection," even though that goodness is really about where the turn eventually leads, not anything that happens immediately. That's exactly what a Q-value captures.

From bandits to states: Q-values

A Q-value answers a more useful question than "what reward did this action just give me?" — it answers "how good is this action, counting everything it eventually leads to?" Q-learning learns these values with a simple, repeated update: take an action, see the real reward and the real next state, and nudge the current Q-value estimate toward reward + (the best Q-value available from the next state). That's genuinely similar to Chapter 4.3's gradient descent — repeatedly nudge an estimate toward a target, a little at a time — except the "target" here is built from the agent's own current (still-imperfect) guesses about the future, not a fixed label.

✓ What you can now explain to someone else
  • A Q-value, Q(s, a), estimates the total future reward of taking action a in state s — not just the immediate reward.
  • Q-learning updates its Q-values by nudging Q(s, a) toward the actual reward received plus the best Q-value available from the resulting next state.
  • This bootstrapped update is similar in spirit to gradient descent (Chapter 4.3) — repeatedly nudging an estimate toward a target — except the target is built from the agent's own evolving estimates, not a fixed external label.
  • Q-learning can converge to sensible values even from a completely random exploration policy, as long as it experiences enough real state-action-reward transitions.

Part XII · Chapter 12.3

Policy Gradients & Exploration/Exploitation

Chapter 12.2's Q-learning learns a table of values, then picks whichever action scores highest. Policy gradients skip the table entirely and adjust action probabilities directly. Both approaches also share a problem this chapter names explicitly: deciding when to try something new versus sticking with what's already worked.

🎤 Frame of reference

A stand-up comedian doesn't assign each joke a precise numeric rating before deciding what to perform. They just notice: that joke got a big laugh — tell it more often. That one fell flat — tell it less. Over many shows, the mix of jokes they actually perform shifts toward whatever's been landing, without ever writing down an exact score for each one. That's the spirit of a policy gradient: adjust how often you do something, directly, based on how it went.

Policy gradients: adjust probabilities directly

A policy gradient method keeps a learnable score for each action, converts those scores into probabilities with softmax (Chapter 8.4), and after sampling an action and observing its real reward, nudges that action's score up or down depending on whether the reward beat what was typically expected. Chapter 9.4's RLHF policy update was already a real example of exactly this — reward-model scores nudging an LLM's token probabilities — applied there to text generation instead of a small, fixed set of actions.

Exploration vs. exploitation

Chapter 12.1's bandit already raised this tension without naming it: always picking whatever action currently looks best is exploitation; deliberately trying something else to learn more is exploration. Too much exploitation risks locking onto an action that only looked good due to limited early evidence, never discovering something genuinely better. Too much exploration wastes reward on actions already known to be worse.

● Interactive demo — try it

The same 3 arms, run with pure exploitation (always pick the best estimate so far, starting from zero information) versus pure exploration (always pick randomly), 200 real pulls each. Watch what pure exploitation actually does when it starts with no information at all.

▶ Run 200 pulls: pure exploitation
▶ Run 200 pulls: pure exploration

Click "pure exploitation": with every estimate starting at zero and ties always breaking toward the first arm, it pulls Arm A — and only Arm A — for all 200 pulls, every single time you run it, since it never has a reason to try B or C once A has any estimate at all. That locks in roughly 30% average reward (Arm A's true rate) for the entire run, permanently missing Arm B's 70%. Click "pure exploration": total reward varies pull to pull, but it's spread across all three arms and typically lands noticeably higher than pure exploitation's 30% — not because exploring is always better, but because this particular exploitation strategy has no way to ever discover it's missing something better. Balancing the two — mostly exploiting, but occasionally still exploring — is what real reinforcement learning algorithms are built to do.

  • Chapter 9.4's RLHF was already a real-world example of a policy gradient update, applied to LLM text generation.
  • Exploitation means picking whatever currently looks best; exploration means deliberately trying something else to learn more.
  • Pure exploitation can get permanently stuck on an early, suboptimal choice if it never has a reason to try alternatives — a real risk the demo showed directly, not just a theoretical concern.

Part XII · Chapter 12.4

Deep RL & RLHF Revisited

Chapter 12.2's Q-table and Chapter 12.3's policy both relied on something that quietly assumed the world stays small: one table entry, or one learnable score, per possible state. Deep reinforcement learning is what happens when that assumption breaks — and closing this Part means going back to Chapter 9.4's RLHF, which turns out to already have been deep RL all along, described in different words.

🚕 Frame of reference

A delivery driver working four fixed intersections could reasonably memorize a table: "at intersection 2, turning left is best." A driver working an entire city can't memorize a table entry for every possible combination of intersection, traffic, weather, and time of day — there are simply too many distinct situations to write each one down. Instead, they develop a general sense of good driving that generalizes: faced with an intersection they've never specifically seen before, they still make a reasonable call, based on how it resembles situations they have seen. A neural network gives a reinforcement learning agent exactly that same kind of generalizing sense, in place of a table.

Deep RL: replacing the table with a network

Deep reinforcement learning (deep RL) swaps Chapter 12.2's Q-table, or Chapter 12.3's per-action score, for a neural network that takes a state as input and outputs Q-values or action probabilities directly — the exact same networks from Chapters 6-8, just trained with reinforcement learning's update rules instead of supervised learning's labels. The table approach never actually stopped being correct in principle; it just stops being possible once the number of distinct states grows large enough.

● Interactive demo — try it

A real computation of exactly how large a Q-table would need to be. Slide the grid size for a simple grid-world (4 actions per cell) to see the table stay entirely manageable — then compare against a real calculation for a single 84×84 grayscale image used as the state instead (a typical size for image-based RL), where a table becomes impossible.

Slide up to a 50×50 grid: still only 10,000 table entries, easy to store. But swap the state representation to a single 84×84 grayscale image — a completely realistic input size for image-based RL — and the number of possible distinct states explodes to roughly 1016,993, a number with about 17,000 digits, dwarfing even the ~81-digit estimate of atoms in the observable universe. No table could ever have one entry for every possible image. A neural network doesn't need one — it generalizes from the states it has actually seen to states it hasn't, the same way a CNN (Chapter 7.2) recognizes a cat it's never seen in a photo it's never seen.
