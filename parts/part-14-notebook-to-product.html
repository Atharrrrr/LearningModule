Part XIV · Chapter 14.1

Version Control for ML

Every chapter so far has focused on building and understanding models. This Part shifts to a different question: how do you actually run one of these things reliably, as part of a real product, maintained by a team? It starts with something that sounds solved already — version control — and shows exactly where ordinary git starts to strain.

📸 Frame of reference

Tracking changes to a written recipe is easy: line 3 changed from "bake 20 minutes" to "bake 25 minutes," and that's the entire diff. Now imagine trying to track changes to a dish by diffing a full-resolution photograph of it after every attempt instead. The lighting shifts a little, the plating shifts a little, and suddenly nearly every pixel in the photo looks "different" from the last one — even though the recipe itself barely changed. A trained model's weight file behaves exactly like that photograph, not like the recipe.

Why git struggles specifically with ML artifacts

Git tracks code beautifully because a code change is usually a handful of lines out of many — a small, meaningful diff. A model checkpoint doesn't work that way. Training nudges every single weight by gradient descent (Chapter 4.3), even slightly, every single step — so two checkpoints saved moments apart can differ in nearly every number they contain. There's no small, meaningful diff to store; the "difference" between versions is effectively the whole file, every time.

● Interactive demo — try it

A real toy 20-number "weight file," compared before and after one more real simulated training step (a small random nudge applied to every number) — versus a real 4-line code file where only one line actually changes.

▶ Run one more training step & compare

Click it a few times: the weight file reliably shows 100% of its numbers changed, every single run, while the code file only ever shows 1 of 4 lines changed. That's not a rare edge case — it's the guaranteed, mechanical consequence of gradient descent touching every parameter on every step. Scale the weight file up from 20 numbers to the millions or billions a real model has, and storing a full new copy for every checkpoint, the way git would try to diff it, becomes enormous, fast.

The fix: track a fingerprint, not a diff

Tools built for this problem don't try to diff large files at all. They use hashing (for large files): compute a compact fingerprint from a checkpoint's contents, store the actual file separately (often in cloud storage), and let git track only that fingerprint alongside the code. A commit then records exactly which checkpoint, which dataset version, and which code all belong together — genuine reproducibility, since recreating a past result requires all three matching exactly, not just the code.

Notice that a hash doesn't measure how much two files differ — the fingerprint for one changed number looks just as different as the fingerprint for every number changing. That's fine for its actual job: confirming exactly which version of a large file is meant to go with which commit, not summarizing what changed inside it.

✓ What you can now explain to someone else
  • Git's diffs work well for code because a typical change touches a small handful of lines out of many.
  • A model checkpoint doesn't have that structure — gradient descent nudges every parameter on every training step, so essentially the entire file changes between versions.
  • ML version control tools store large files separately and track a short hash fingerprint of them in git instead of trying to diff them directly.
  • Real reproducibility requires the exact code, data version, and hyperparameters to all be tracked together — any one of them differing can change the result.

Part XIV · Chapter 14.2

Experiment Tracking

Chapter 14.1 covered tracking versions of code, data, and checkpoints. Training a real model is rarely one run — it's dozens or hundreds of runs, each with different settings, and without a systematic record of which settings produced which result, comparing them (or reproducing the good one) becomes genuinely impossible.

🧪 Frame of reference

A scientist who runs the same experiment dozens of times with slightly different settings, but never writes down which setting went with which result, has no way to know which combination actually worked — or to run it again. A proper lab notebook logs every run's exact conditions and outcome side by side, so results can be compared honestly and the best one can be reproduced deliberately, not remembered by guesswork.

Hyperparameters, and why tracking them matters

A hyperparameter is chosen before training starts, not learned during it — learning rate, batch size, network depth. Different choices can produce wildly different results from the exact same model and data. Experiment tracking means logging every run's hyperparameters alongside its actual outcome, so instead of vaguely remembering "the run with the smaller learning rate seemed better," there's a real, searchable record of exactly which setting produced exactly which result.

  • Without a systematic record connecting settings to outcomes, comparing many training runs honestly becomes essentially impossible.

Part XIV · Chapter 14.3

Hardware: CPU vs. GPU vs. TPU

Every layer in every network this book has covered boils down to matrix multiplication (Chapter 2.3). That single fact explains why neural networks are trained on specialized hardware instead of the same processor that runs everything else on a computer.

🍳 Frame of reference

A small team of expert chefs can each handle a complicated, varied dish from start to finish — flexible, but only a few dishes cooking at once. A kitchen with a thousand line cooks, each doing one simple, repetitive task — chop this one vegetable, over and over — looks far less impressive per worker, but if the actual job is "chop ten thousand identical vegetables," that crew of simple workers, all chopping simultaneously, finishes far faster than the small team of experts ever could. Neural network training is overwhelmingly that second kind of job.

Matrix multiplication is enormous, independent, repetitive work

A matrix multiply (Chapter 2.3) inside a layer (Chapter 6.2) is made of enormous numbers of individual multiply-and-add operations — and critically, none of them depends on any other one's result. That's exactly the condition parallelization needs: pieces of work that can all run at the very same time, on separate workers, instead of one after another.

● Interactive demo — try it

A real operation-count and time estimate for multiplying two N×N matrices (N³ multiply-add operations — the standard, verifiable cost of matrix multiplication), comparing an 8-core CPU against a 4096-core GPU, both running at the same illustrative 1 billion operations/second per core.

Slide N up to 4096 — a realistic hidden-layer size for a real Transformer (Chapter 8.5) — and one single matrix multiply takes the 8-core CPU roughly 8.6 seconds, versus about 0.017 seconds on the 4096-core GPU. A real training run repeats this exact operation billions of times across every layer and every step. At small N the absolute gap barely matters; at real neural-network scale, it's the difference between a training run finishing in hours versus not finishing in any practical amount of time at all.

GPUs and TPUs: built for exactly this shape of work

A CPU has only a handful of powerful, flexible cores, built to handle complex, varied, sequential logic well — exactly wrong for a job made of millions of identical, independent operations. A GPU (Graphics Processing Unit) has thousands of much simpler cores, originally built to color millions of independent pixels at once for rendering graphics — a workload with the identical "massively parallel, simple, repetitive" shape neural network training needs. A TPU (Tensor Processing Unit) pushes specialization further still: hardware built by Google specifically for tensor operations, trading away general-purpose flexibility entirely in exchange for even more speed at exactly this one shape of computation.

✓ What you can now explain to someone else
  • Every layer's computation is a matrix multiplication, made of enormous numbers of independent multiply-add operations.
  • Independent operations can be parallelized — computed simultaneously on separate workers instead of one after another.
  • A GPU has thousands of simple cores, well suited to this kind of massively parallel, repetitive work — the same shape of workload it was originally built for in graphics rendering.
  • A TPU specializes even further, purpose-built for neural network tensor operations at the cost of general-purpose flexibility.

Part XIV · Chapter 14.4

Distributed & Parallel Training

Chapter 14.3 covered parallelism inside one GPU — thousands of cores splitting up a single matrix multiply. This chapter covers parallelism across many GPUs or machines at once, splitting up the training workload itself, needed once a single GPU's compute or memory simply isn't enough.

🏭 Frame of reference

One kitchen with a thousand cooks (Chapter 14.3's parallel cores) can only get so big before adding more cooks stops helping. A catering company facing an enormous order instead opens several separate kitchens, each preparing an equal fraction of the order independently, then combines everyone's output at the end. That's a completely different kind of scaling — not more workers in one kitchen, but more kitchens entirely, each doing a share of the job in parallel.

Data parallelism: split the batch, average the gradients

Data parallelism splits the data, not the model: each GPU gets a slice of the batch and an identical copy of the current weights, computes its own gradients via backpropagation (Chapter 6.4), and then all the GPUs' gradients get averaged together into one combined gradient before a single optimizer step (Chapter 6.5) updates every copy identically. Because a loss averaged over a batch is just the average of each example's individual loss, averaging gradients computed on separate slices produces mathematically the exact same result as computing the gradient on the whole batch at once — not an approximation, an identity.

● Interactive demo — try it

A real toy model (y = w·x) and 4 real data points. Compute the gradient the ordinary way, on the full batch — then split the same 4 points across 2 simulated GPUs, compute each GPU's gradient separately, and average them. Watch the two results match exactly.

▶ Compute both ways

Both routes land on exactly −18.0000 — running 2 GPUs on half the data each, then averaging, gives precisely the same gradient as one device processing the whole batch. That exact equivalence is what makes data parallelism trustworthy: splitting the work across more devices doesn't change what's actually being computed, it only changes how quickly it gets computed, since each GPU's slice can be processed simultaneously.

Model parallelism: split the model instead

Data parallelism assumes the whole model fits on every GPU — fine until the model itself is the problem. A modern LLM (Chapter 9.2) with billions of parameters may simply not fit in one GPU's memory at all, no matter how the data is split. Model parallelism is the other option: split the model's own layers across multiple GPUs — one device holds the first several layers, the next device holds the following several, and data flows from one device to the next in sequence. It solves a memory problem data parallelism can't, at the cost of GPUs now having to wait on each other in sequence rather than working on fully independent slices at once.

✓ What you can now explain to someone else
  • Data parallelism splits a batch across GPUs, each with a full copy of the model, then averages their gradients into one update.
  • Averaging gradients from equal-sized data splits produces exactly the same result as computing the gradient on the full batch — a mathematical identity, not an approximation.
  • Model parallelism splits the model's own layers across devices instead, needed when the model itself doesn't fit in one GPU's memory.
  • Data parallelism trades nothing in correctness for speed; model parallelism trades some communication overhead for the memory capacity to train models too large for a single device.

Part XIV · Chapter 14.5

Serving a Model

Every chapter on training has assumed the goal is producing a good model. Once that model exists, a different job begins: answering real, live requests from real users, fast enough and cheaply enough to actually run as a product.

🚕 Frame of reference

A taxi leaves the instant one passenger is ready — the fastest possible trip for that one rider, but the vehicle's capacity for extra passengers goes unused. A bus waits a few minutes to collect a full load of passengers before departing — any one passenger waits a little longer to leave, but the vehicle carries far more people per trip. Serving a trained model faces exactly this choice, request by request, all day long.

Inference: using the model, not training it

Inference is running a finished, trained model forward to get a prediction — the same forward pass (Chapter 6.2) used throughout training, just without backpropagation or a weight update afterward. Serving inference at scale means balancing two different things: latency, how long one request waits for its answer, and throughput, how many requests the whole system handles per second.

Batching requests: the same tradeoff as the taxi and the bus

Chapter 6.6's training batches group examples together for efficiency; serving does the same thing to incoming requests. Answering each request the instant it arrives (batch size 1) minimizes that request's own latency, but wastes most of a GPU's parallel capacity (Chapter 14.3) — it can process many requests in the same single forward pass for barely more time than processing one. Waiting briefly to collect several requests into one batch adds a small amount of latency, but multiplies how many requests get answered per second.

● Interactive demo — try it

A real computation: one forward pass takes a fixed 50ms, regardless of batch size, up to hardware capacity (Chapter 14.3's parallelism). Batching waits up to 20ms to collect a fuller batch first. Compare real latency and throughput at different batch sizes.

Slide from batch size 1 to 32: latency only rises from 50ms to 70ms — a small, fixed wait added once — while throughput climbs from 20 requests/second all the way to over 450. A tiny, bounded increase in each request's wait buys a roughly 23× increase in how many requests the whole system can serve per second. That's the entire tradeoff real serving systems are built to balance, computed directly rather than just described.

✓ What you can now explain to someone else
  • Inference is running a trained model forward on a new input, without backpropagation or a weight update — a different job from training.
  • Latency measures one request's wait time; throughput measures the whole system's total request-handling capacity.
  • Serving one request at a time minimizes its individual latency but wastes most of a GPU's parallel capacity.
  • Batching several requests together adds a small, bounded amount of latency but multiplies throughput, since the same forward-pass time serves many more requests at once.

Part XIV · Chapter 14.6

Containers & Reproducibility

Chapter 14.1 established that reproducibility needs the exact code, data version, and hyperparameters that produced a result. There's a fourth piece easy to overlook: the exact software environment those ran in. Get that one wrong, and even identical code, data, and settings can behave differently.

🍞 Frame of reference

A recipe that works perfectly in one kitchen can come out differently in another — a different oven that runs a little hotter, a different altitude, a different brand of flour. The written recipe is identical; the environment it's executed in isn't, and that's enough to change the result. Software has the exact same problem: identical code can behave differently depending on which library versions, operating system, or drivers happen to be installed on the machine running it.

Identical code, different environment, different result

This isn't hypothetical — it can be demonstrated with plain arithmetic. Floating-point numbers (Chapter 2.1) aren't perfectly associative: adding the exact same set of numbers in a different order can produce a subtly different final result, due to rounding at each intermediate step. Different library versions, hardware, or execution paths can sum numbers internally in different orders — meaning "the same" mathematical operation can genuinely return a different real number depending on the environment it ran in.

● Interactive demo — try it

The exact same 12 real numbers, summed in two different real orders — forward and backward — nothing else different.

▶ Sum both ways

The two sums genuinely differ, in the real floating-point arithmetic your browser just performed — not a rounding display artifact, an actual different number. Now imagine that difference happening not because someone reordered a list on purpose, but because a training run happened on a machine with a slightly different library version internally summing numbers in a different order. Two "identical" runs, silently producing different results, for a reason that has nothing to do with the code or the data at all.

Containers: packaging the whole environment

A container solves this by packaging code together with its entire software environment — every library at an exact version, every system dependency — into one portable unit that behaves identically no matter which machine runs it. Building one properly means pinning (dependency versions): writing down the exact version of every library used, not just "the newest one available," so the environment itself becomes part of the reproducible record alongside Chapter 14.1's code, data, and hyperparameters.

✓ What you can now explain to someone else
  • Identical code, data, and hyperparameters can still produce different results if the underlying software environment differs.
  • Floating-point addition isn't perfectly associative — summing the same real numbers in a different order can produce a genuinely different result, a real, demonstrable fact, not a hypothetical.
  • A container packages code together with its exact software dependencies into one portable, self-contained unit.
  • Pinning dependency versions — recording the exact version of every library used — is what makes the environment itself part of a reproducible record.
