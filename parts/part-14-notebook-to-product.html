Part XIV · Chapter 14.1

Version Control for ML

Every chapter so far has focused on building and understanding models. This Part shifts to a different question: how do you actually run one of these things reliably, as part of a real product, maintained by a team? It starts with something that sounds solved already — version control — and shows exactly where ordinary git starts to strain.

📸 Frame of reference

Tracking changes to a written recipe is easy: line 3 changed from "bake 20 minutes" to "bake 25 minutes," and that's the entire diff. Now imagine trying to track changes to a dish by diffing a full-resolution photograph of it after every attempt instead. The lighting shifts a little, the plating shifts a little, and suddenly nearly every pixel in the photo looks "different" from the last one — even though the recipe itself barely changed. A trained model's weight file behaves exactly like that photograph, not like the recipe.

Why git struggles specifically with ML artifacts

Git tracks code beautifully because a code change is usually a handful of lines out of many — a small, meaningful diff. A model checkpoint doesn't work that way. Training nudges every single weight by gradient descent (Chapter 4.3), even slightly, every single step — so two checkpoints saved moments apart can differ in nearly every number they contain. There's no small, meaningful diff to store; the "difference" between versions is effectively the whole file, every time.

● Interactive demo — try it

A real toy 20-number "weight file," compared before and after one more real simulated training step (a small random nudge applied to every number) — versus a real 4-line code file where only one line actually changes.

▶ Run one more training step & compare

Click it a few times: the weight file reliably shows 100% of its numbers changed, every single run, while the code file only ever shows 1 of 4 lines changed. That's not a rare edge case — it's the guaranteed, mechanical consequence of gradient descent touching every parameter on every step. Scale the weight file up from 20 numbers to the millions or billions a real model has, and storing a full new copy for every checkpoint, the way git would try to diff it, becomes enormous, fast.

The fix: track a fingerprint, not a diff

Tools built for this problem don't try to diff large files at all. They use hashing (for large files): compute a compact fingerprint from a checkpoint's contents, store the actual file separately (often in cloud storage), and let git track only that fingerprint alongside the code. A commit then records exactly which checkpoint, which dataset version, and which code all belong together — genuine reproducibility, since recreating a past result requires all three matching exactly, not just the code.

Notice that a hash doesn't measure how much two files differ — the fingerprint for one changed number looks just as different as the fingerprint for every number changing. That's fine for its actual job: confirming exactly which version of a large file is meant to go with which commit, not summarizing what changed inside it.

✓ What you can now explain to someone else
  • Git's diffs work well for code because a typical change touches a small handful of lines out of many.
  • A model checkpoint doesn't have that structure — gradient descent nudges every parameter on every training step, so essentially the entire file changes between versions.
  • ML version control tools store large files separately and track a short hash fingerprint of them in git instead of trying to diff them directly.
  • Real reproducibility requires the exact code, data version, and hyperparameters to all be tracked together — any one of them differing can change the result.

Part XIV · Chapter 14.2

Experiment Tracking

Chapter 14.1 covered tracking versions of code, data, and checkpoints. Training a real model is rarely one run — it's dozens or hundreds of runs, each with different settings, and without a systematic record of which settings produced which result, comparing them (or reproducing the good one) becomes genuinely impossible.

🧪 Frame of reference

A scientist who runs the same experiment dozens of times with slightly different settings, but never writes down which setting went with which result, has no way to know which combination actually worked — or to run it again. A proper lab notebook logs every run's exact conditions and outcome side by side, so results can be compared honestly and the best one can be reproduced deliberately, not remembered by guesswork.

Hyperparameters, and why tracking them matters

A hyperparameter is chosen before training starts, not learned during it — learning rate, batch size, network depth. Different choices can produce wildly different results from the exact same model and data. Experiment tracking means logging every run's hyperparameters alongside its actual outcome, so instead of vaguely remembering "the run with the smaller learning rate seemed better," there's a real, searchable record of exactly which setting produced exactly which result.

✓ What you can now explain to someone else
  • A hyperparameter is chosen before training starts, unlike a parameter, which is learned from data during training.
  • Experiment tracking logs every run's hyperparameters, resulting metrics, and checkpoint together, so runs can be compared and reproduced.
  • Small changes to a single hyperparameter, like learning rate, can produce dramatically different real outcomes — from near-perfect convergence to complete divergence.
  • Without a systematic record connecting settings to outcomes, comparing many training runs honestly becomes essentially impossible.

Part XIV · Chapter 14.3

Hardware: CPU vs. GPU vs. TPU

Every layer in every network this book has covered boils down to matrix multiplication (Chapter 2.3). That single fact explains why neural networks are trained on specialized hardware instead of the same processor that runs everything else on a computer.

🍳 Frame of reference

A small team of expert chefs can each handle a complicated, varied dish from start to finish — flexible, but only a few dishes cooking at once. A kitchen with a thousand line cooks, each doing one simple, repetitive task — chop this one vegetable, over and over — looks far less impressive per worker, but if the actual job is "chop ten thousand identical vegetables," that crew of simple workers, all chopping simultaneously, finishes far faster than the small team of experts ever could. Neural network training is overwhelmingly that second kind of job.

Matrix multiplication is enormous, independent, repetitive work

A matrix multiply (Chapter 2.3) inside a layer (Chapter 6.2) is made of enormous numbers of individual multiply-and-add operations — and critically, none of them depends on any other one's result. That's exactly the condition parallelization needs: pieces of work that can all run at the very same time, on separate workers, instead of one after another.

● Interactive demo — try it

A real operation-count and time estimate for multiplying two N×N matrices (N³ multiply-add operations — the standard, verifiable cost of matrix multiplication), comparing an 8-core CPU against a 4096-core GPU, both running at the same illustrative 1 billion operations/second per core.

Slide N up to 4096 — a realistic hidden-layer size for a real Transformer (Chapter 8.5) — and one single matrix multiply takes the 8-core CPU roughly 8.6 seconds, versus about 0.017 seconds on the 4096-core GPU. A real training run repeats this exact operation billions of times across every layer and every step. At small N the absolute gap barely matters; at real neural-network scale, it's the difference between a training run finishing in hours versus not finishing in any practical amount of time at all.

GPUs and TPUs: built for exactly this shape of work

A CPU has only a handful of powerful, flexible cores, built to handle complex, varied, sequential logic well — exactly wrong for a job made of millions of identical, independent operations. A GPU (Graphics Processing Unit) has thousands of much simpler cores, originally built to color millions of independent pixels at once for rendering graphics — a workload with the identical "massively parallel, simple, repetitive" shape neural network training needs. A TPU (Tensor Processing Unit) pushes specialization further still: hardware built by Google specifically for tensor operations, trading away general-purpose flexibility entirely in exchange for even more speed at exactly this one shape of computation.

✓ What you can now explain to someone else
  • Every layer's computation is a matrix multiplication, made of enormous numbers of independent multiply-add operations.
  • Independent operations can be parallelized — computed simultaneously on separate workers instead of one after another.
  • A GPU has thousands of simple cores, well suited to this kind of massively parallel, repetitive work — the same shape of workload it was originally built for in graphics rendering.
  • A TPU specializes even further, purpose-built for neural network tensor operations at the cost of general-purpose flexibility.
