Part VIII · Chapter 8.1

Why Order Matters

Every model so far has treated its input as a fixed bundle of features, or a fixed grid of pixels. Language, audio, and time series break that assumption in a specific way: the exact same pieces, rearranged, can mean something completely different.

🐕 Frame of reference

"The dog bit the man" and "the man bit the dog" use the exact same five words. One is a mundane news brief; the other is a much stranger headline. Nothing about the set of words changed — only their order — and yet the meaning flipped entirely. Any representation that throws away order would see these two sentences as identical.

When "what's present" isn't enough

A bag of words is exactly Chapter 3.4's feature-engineering idea applied to text: turn a sentence into a vector (Chapter 2.2) of word counts. It's a genuinely useful representation for plenty of problems — but it has a specific blind spot.

The two vectors come out exactly the same, every single time — this isn't a coincidence or a bug, it's the direct, guaranteed consequence of a representation that only counts which words showed up, never where. Anything downstream that looks only at these vectors has no way to ever recover which sentence bit whom.

Sequences need position, not just presence

Text, audio, and time series share this trait: they're sequences — ordered lists where position carries real information, and where the length itself varies from example to example (a 3-word sentence and a 30-word sentence are both valid sentences). Chapter 6's networks expect a fixed number of inputs; a plain feedforward layer has no built-in way to handle "sometimes 3 words, sometimes 30," and no built-in notion of position at all. The rest of this Part builds architectures designed around exactly that gap: RNNs (Chapter 8.2) that process one element at a time while carrying forward a memory of what came before, and attention (Chapter 8.4) that looks at an entire sequence at once while explicitly tracking each element's position.

✓ What you can now explain to someone else
  • A bag-of-words vector counts word occurrences but discards order entirely — two sentences with the same words in different orders produce an identical vector.
  • Order can carry essential meaning ("the dog bit the man" vs. "the man bit the dog"), which an order-blind representation can never recover.
  • Sequences (text, audio, time series) also vary in length, which plain fixed-input networks don't naturally handle.
  • The rest of this Part builds architectures — RNNs, then attention and Transformers — designed specifically around order and variable length.

Part VIII · Chapter 8.2

RNNs & Their Limits

Chapter 8.1 identified the gap: sequences vary in length and depend on order. A recurrent neural network fills that gap with an idea that's almost embarrassingly simple — read one element at a time, and carry a running memory forward.

📖 Frame of reference

Reading a sentence one word at a time. By the time you reach the tenth word, you're not replaying the previous nine verbatim in your head — you're carrying forward a condensed "understanding so far," updated after every new word. That running, updated summary is exactly what a recurrent network carries from one step to the next.

The same neuron, applied one step at a time

A recurrent neural network (RNN) keeps a hidden state that gets updated at every step of the sequence: h_t = sigmoid(w_h·h_(t-1) + w_x·x_t + b) — Chapter 6.1's neuron formula, applied repeatedly, using the exact same weights every single step. That's weight sharing again, just like Chapter 7.2's kernel reused across every position in an image — except an RNN reuses its weights across time instead of space.

Watch the hidden state rise sharply whenever the keyword appears, then decay a little on steps without it — a real, computed running memory of "how recently, and how often, has the keyword shown up?" No fixed-length input was ever needed; the same update rule just keeps running for as many steps as the sequence has.

The limit: memory fades over long sequences

That repeated multiplication is also exactly Chapter 6.7's vanishing-gradient setup, wearing different clothes. Unrolled across T steps, an RNN is mathematically just as deep as a T-layer feedforward network. Backpropagating an error back to an early step multiplies through T local derivatives — and if each one is a fraction less than 1 (sigmoid's again tops out at 0.25), that gradient vanishes exactly as fast as it did across depth.

● Interactive demo — try it

Chapter 6.7's exact math, relabeled: gradient magnitude versus how many steps back in the sequence an event happened.

By 10-20 steps back, the gradient has collapsed to essentially nothing — an event early in a long sequence barely influences training at all, no matter how important it actually was to the outcome. Plain RNNs genuinely struggle to connect distant parts of a long sequence. Fixing this properly is exactly what Chapter 8.4's attention mechanism was built to do.

✓ What you can now explain to someone else
  • An RNN processes a sequence one element at a time, updating a hidden state with the same weights at every step — weight sharing across time.
  • The hidden state is a running memory: a summary of everything processed so far, not a verbatim record.
  • RNNs naturally handle variable-length sequences, since the same update rule just runs for as many steps as needed.
  • Unrolled over many steps, an RNN suffers the same vanishing-gradient problem as a very deep network (Chapter 6.7), making long-range dependencies hard to learn.

Part VIII · Chapter 8.3

Word Embeddings (word2vec)

Chapter 8.1's bag-of-words fixed nothing about a deeper problem: one-hot encoding (Chapter 3.4) treats "cat" and "dog" as just as unrelated as "cat" and "airplane." Every word is equally distant from every other word — no notion of meaning at all.

🗺️ Frame of reference

A list of city names carries no sense of geography — "Paris" and "Berlin" are just two strings, no closer to each other than "Paris" and "Tokyo." Put those same cities on an actual map, though, and distance becomes meaningful: Paris and Berlin land near each other, Tokyo lands far away, and the map even encodes relationships — the direction from Paris to France's other cities looks similar to the direction from Berlin to Germany's. Word embeddings are exactly this kind of map, built for words instead of cities.

From arbitrary symbols to a meaningful map

A word embedding replaces one-hot encoding's giant, meaningless sparse vector with a short, dense one — typically 50 to 300 numbers — positioned so that words used in similar contexts land near each other. word2vec is the classic way to learn these: train a model to predict a word from its neighbors across huge amounts of real text, and words that keep similar company end up with similar vectors, entirely from that training signal — nobody hand-labels "cat is similar to dog."

The arithmetic lands exactly on queen's real coordinates — this toy example was built to make that exact, but the same kind of relationship shows up reliably in real word2vec embeddings trained on ordinary text, with nobody ever telling the model "gender" or "royalty" are meaningful directions. It's also worth noticing the dog/cat cluster sits far from the king/queen/man/woman cluster — distance in embedding space tracks meaning, exactly the property one-hot encoding never had.

Why this fixes bag-of-words too

Chapter 8.1's bag-of-words problem was about losing order; one-hot's problem is about losing similarity. Embeddings solve the second one directly, and set up everything from here on: instead of feeding one-hot vectors into a sequence model, feed each word's embedding — letting the model reason with "close in meaning" as an actual geometric fact, not something it has to rediscover from scratch every time.

✓ What you can now explain to someone else
  • One-hot encoding treats every word as equally distant from every other — no notion of meaning.
  • A word embedding is a dense, learned vector positioned so semantically similar words land close together.
  • word2vec learns embeddings by training on predicting words from their context across huge amounts of text, with no manual similarity labeling.
  • Embedding arithmetic (king − man + woman ≈ queen) reveals that meaningful relationships can become consistent directions in the vector space.

Part VIII · Chapter 8.4

Attention

"The trophy doesn't fit in the suitcase because it is too big." What does "it" refer to? Now swap one word — "...because it is too small" — and the answer flips entirely. Resolving that pronoun means directly comparing "it" against every candidate noun in the sentence, no matter how far back they are. That direct comparison, done with real numbers, is attention.

🏆 Frame of reference

Reading that sentence, you don't process "it" by slowly replaying every word that came before, hoping enough survived in your memory. You directly glance back at "trophy" and "suitcase" — the only two real candidates — and weigh each one's relevance given the rest of the sentence. "Big" tips the weighing toward "trophy" (the thing too big to fit); "small" tips it toward "suitcase" (the thing too small to hold it). Attention is that direct glance-and-weigh, turned into arithmetic.

Query, Key, Value: asking, offering, answering

Attention gives every position three roles. A query: what is this position trying to figure out (for "it," roughly "what noun am I standing in for?"). A key: what does each candidate position offer to be matched against (what "trophy" and "suitcase" each represent). A value: the actual content contributed if chosen. Compare the query against every key with a dot product (Chapter 2.4) — a bigger dot product means a better match. Turn those scores into weights that sum to 1 with softmax, then take a weighted sum of every value using those weights. No step-by-step chain, no distance penalty — every position is one dot product away from every other.

Flip the toggle and the softmax weights flip with it — 94.4% trophy in the "big" case, 94.4% suitcase in the "small" case — computed from the exact same mechanism, just a different query. Nothing here traveled step by step through a hidden state the way Chapter 8.2's RNN would; "it" compared itself directly against both candidates in one shot, regardless of how many words separated them.

The ÷√2 scaling matters more than it looks: as vectors get longer (real Transformers use hundreds of dimensions, not 2), raw dot products grow larger purely from having more terms to sum, which would push softmax toward an extreme 100%/0% split regardless of how genuinely different the scores are. Dividing by the square root of the dimension keeps scores in a range where softmax can still express real uncertainty.

✓ What you can now explain to someone else
  • Attention compares a query against every key with a dot product, converts those scores to weights with softmax, then takes a weighted sum of values.
  • Unlike an RNN, attention lets any two positions interact directly in one step, with no distance penalty and no vanishing-gradient chain to survive.
  • Softmax turns a list of raw scores into probabilities summing to 1, generalizing sigmoid to more than two options.
  • Scores are scaled down (by the square root of the vector dimension) before softmax, to keep the weights from saturating to an extreme split as dimensions grow.

Part VIII · Chapter 8.5

The Transformer, Piece by Piece

You've now built every individual part: embeddings (8.3), attention (8.4), feedforward layers (6.2), residual connections (7.5). A Transformer doesn't add much that's genuinely new — it bolts these exact parts together into one block, fixes one gap attention leaves open, and stacks the result.

🔧 Frame of reference

You've built an engine, a transmission, and a chassis, each on its own workbench. None of them is a car by itself. A Transformer block is the assembly step — bolting parts you already understand into one running machine, in a specific order, with a couple of connectors that make sure nothing gets lost between stages.

The gap attention leaves open: position

Look closely at Chapter 8.4's attention formula: a dot product between a query and a key doesn't reference position anywhere. Swap two input positions, and attention produces the exact same set of comparisons, just relabeled — it genuinely cannot tell "trophy" at position 1 from "trophy" at position 5. That reopens Chapter 8.1's entire problem, inside the very mechanism built to solve it.

Positional encoding patches this directly: add a position-dependent vector to each token's embedding before attention ever runs. The standard recipe uses sine and cosine waves at different frequencies, so every position gets a unique numeric signature: PE(pos, 2i) = sin(pos / 10000^(2i/d)), PE(pos, 2i+1) = cos(pos / 10000^(2i/d)).

Add this vector to a token's embedding, and "trophy" at position 1 and "trophy" at position 5 arrive at attention as genuinely different vectors, breaking the symmetry a dot product alone could never see.

Multi-head attention: several relationships at once

One attention computation learns one notion of "relevant." Multi-head attention runs several in parallel — one head might specialize in resolving references like Chapter 8.4's "it," another might track which adjective modifies which noun. Each head produces its own weighted output; concatenating them gives the block more than one lens on the same sequence at once.

Feedforward, residuals, and stacking

After attention mixes information across positions, a plain feedforward layer (Chapter 6.2) — the same small MLP, applied identically at every position — processes each position's result independently, adding capacity attention itself doesn't provide. Both the attention step and the feedforward step are wrapped in a residual connection (Chapter 7.5): the block's input is added back to its output, keeping gradients healthy even when many blocks are stacked. A normalization step (closely related to Chapter 6.8's BatchNorm, but normalizing across each position's own features rather than across a batch) keeps signal scale stable between blocks. Stack several of these blocks — each one attention, then feedforward, each wrapped in residual-and-normalize — and you have a Transformer.

Part VIII · Chapter 8.6

Meet SuperGlue

Chapter 7.3's descriptor matching compared each keypoint to its closest match in isolation — one pair at a time, blind to everyone else. SuperGlue asks a sharper question: what if keypoints could compare notes with each other before anyone commits to a match?

🏫 Frame of reference

Matching students in this year's class photo to last year's, using only face crops compared one-on-one — no memory of who's standing near whom. Twins, or two students with similar haircuts, get confused constantly, because each face is judged in total isolation. Now let the students compare notes: "I'm standing between the two tallest kids in both photos" — suddenly ambiguous individual matches resolve using the relationships between everyone, not just isolated appearances.

Keypoints as a graph, attention as message-passing

Treat every keypoint (Chapter 7.3, or Chapter 7.8's SuperPoint) as a node in a graph. SuperGlue runs attention (Chapter 8.4) as message-passing on that graph: self-attention lets keypoints within the same image exchange context with each other; cross-attention lets keypoints in image A directly attend to keypoints in image B. Stack several rounds of this — a Transformer-style block (Chapter 8.5) — and by the time matching actually happens, every keypoint's representation already encodes its relationships to everything else, not just its own isolated descriptor.

By descriptor distance alone, P2 looks closer to Q3 and P3 looks closer to Q2 — nearest-neighbor matching (Chapter 7.3) would confidently swap them. The true correspondence is actually P2↔Q2 and P3↔Q3.

At λ = 0 (descriptors only, exactly Chapter 7.3's approach), the swapped, wrong assignment wins — its descriptor distances are just slightly smaller. Raise λ past about 1.6 and the correct assignment takes over: the swapped pairing badly distorts the real geometric relationships between all three points, while the correct pairing preserves them perfectly. SuperGlue's attention learns something in this spirit automatically — using every keypoint's relationship to every other keypoint, not just one isolated descriptor comparison.

Real SuperGlue adds one more real detail worth knowing: not every keypoint has a match at all (an object might be visible in one image and occluded in the other). Rather than forcing every point to match something, SuperGlue adds a "dustbin" — an explicit option to match nothing — and solves the whole assignment as an optimal transport problem, letting points legitimately go unmatched instead of forcing a wrong pairing.

✓ What you can now explain to someone else
  • Nearest-neighbor descriptor matching (Chapter 7.3) judges each keypoint pair in isolation, with no awareness of other keypoints.
  • SuperGlue treats keypoints as nodes in a graph, using self-attention within each image and cross-attention between images to share context before matching.
  • Incorporating relationships between keypoints (like preserved geometric consistency) can correct matches that descriptor similarity alone gets wrong.
  • SuperGlue includes a "dustbin" option so keypoints with no true match aren't forced into a wrong pairing.
