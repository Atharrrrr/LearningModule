Part VIII · Chapter 8.1

Why Order Matters

Every model so far has treated its input as a fixed bundle of features, or a fixed grid of pixels. Language, audio, and time series break that assumption in a specific way: the exact same pieces, rearranged, can mean something completely different.

🐕 Frame of reference

"The dog bit the man" and "the man bit the dog" use the exact same five words. One is a mundane news brief; the other is a much stranger headline. Nothing about the set of words changed — only their order — and yet the meaning flipped entirely. Any representation that throws away order would see these two sentences as identical.

When "what's present" isn't enough

A bag of words is exactly Chapter 3.4's feature-engineering idea applied to text: turn a sentence into a vector (Chapter 2.2) of word counts. It's a genuinely useful representation for plenty of problems — but it has a specific blind spot.

The two vectors come out exactly the same, every single time — this isn't a coincidence or a bug, it's the direct, guaranteed consequence of a representation that only counts which words showed up, never where. Anything downstream that looks only at these vectors has no way to ever recover which sentence bit whom.

Sequences need position, not just presence

Text, audio, and time series share this trait: they're sequences — ordered lists where position carries real information, and where the length itself varies from example to example (a 3-word sentence and a 30-word sentence are both valid sentences). Chapter 6's networks expect a fixed number of inputs; a plain feedforward layer has no built-in way to handle "sometimes 3 words, sometimes 30," and no built-in notion of position at all. The rest of this Part builds architectures designed around exactly that gap: RNNs (Chapter 8.2) that process one element at a time while carrying forward a memory of what came before, and attention (Chapter 8.4) that looks at an entire sequence at once while explicitly tracking each element's position.

✓ What you can now explain to someone else
  • A bag-of-words vector counts word occurrences but discards order entirely — two sentences with the same words in different orders produce an identical vector.
  • Order can carry essential meaning ("the dog bit the man" vs. "the man bit the dog"), which an order-blind representation can never recover.
  • Sequences (text, audio, time series) also vary in length, which plain fixed-input networks don't naturally handle.
  • The rest of this Part builds architectures — RNNs, then attention and Transformers — designed specifically around order and variable length.

Part VIII · Chapter 8.2

RNNs & Their Limits

Chapter 8.1 identified the gap: sequences vary in length and depend on order. A recurrent neural network fills that gap with an idea that's almost embarrassingly simple — read one element at a time, and carry a running memory forward.

📖 Frame of reference

Reading a sentence one word at a time. By the time you reach the tenth word, you're not replaying the previous nine verbatim in your head — you're carrying forward a condensed "understanding so far," updated after every new word. That running, updated summary is exactly what a recurrent network carries from one step to the next.

The same neuron, applied one step at a time

A recurrent neural network (RNN) keeps a hidden state that gets updated at every step of the sequence: h_t = sigmoid(w_h·h_(t-1) + w_x·x_t + b) — Chapter 6.1's neuron formula, applied repeatedly, using the exact same weights every single step. That's weight sharing again, just like Chapter 7.2's kernel reused across every position in an image — except an RNN reuses its weights across time instead of space.

Watch the hidden state rise sharply whenever the keyword appears, then decay a little on steps without it — a real, computed running memory of "how recently, and how often, has the keyword shown up?" No fixed-length input was ever needed; the same update rule just keeps running for as many steps as the sequence has.

The limit: memory fades over long sequences

That repeated multiplication is also exactly Chapter 6.7's vanishing-gradient setup, wearing different clothes. Unrolled across T steps, an RNN is mathematically just as deep as a T-layer feedforward network. Backpropagating an error back to an early step multiplies through T local derivatives — and if each one is a fraction less than 1 (sigmoid's again tops out at 0.25), that gradient vanishes exactly as fast as it did across depth.

● Interactive demo — try it

Chapter 6.7's exact math, relabeled: gradient magnitude versus how many steps back in the sequence an event happened.

By 10-20 steps back, the gradient has collapsed to essentially nothing — an event early in a long sequence barely influences training at all, no matter how important it actually was to the outcome. Plain RNNs genuinely struggle to connect distant parts of a long sequence. Fixing this properly is exactly what Chapter 8.4's attention mechanism was built to do.

✓ What you can now explain to someone else
  • An RNN processes a sequence one element at a time, updating a hidden state with the same weights at every step — weight sharing across time.
  • The hidden state is a running memory: a summary of everything processed so far, not a verbatim record.
  • RNNs naturally handle variable-length sequences, since the same update rule just runs for as many steps as needed.
  • Unrolled over many steps, an RNN suffers the same vanishing-gradient problem as a very deep network (Chapter 6.7), making long-range dependencies hard to learn.
