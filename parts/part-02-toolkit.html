Part II · Chapter 2.1

Numbers, Variables & Functions

Chapter 1.3 said a model is just a function with adjustable knobs. This chapter makes "function" precise — the last piece of vocabulary you need before the real math starts.

🥤 Frame of reference

Think of a vending machine. You press a specific button, and it always gives you the exact same snack for that button, every single time — never something different on a whim. The button you press is like an input. The snack that comes out is the output. And the machine's fixed wiring — which button leads to which snack — never changes between uses. A function is that machine. A variable is just the label on the button before you've decided which one to press.

Variables: names that can change

A variable is just a name — x, hours, score — that stands in for a number. The whole point is that the number it stands for can change: hours might be 3 for one student and 7 for another, while the name "hours" stays the same. You already used variables without the formal name back in Chapter 1.3, when the exam-score line used m and b as the knobs you could turn.

Functions: a rule, applied consistently

A function is a rule that turns an input into an output — always the same output for the same input, like the vending machine. It's usually written f(x), read as "f of x": the function named f, applied to the input x. If f(x) = 2x + 1, that's the whole rule — double the input, then add one. Plugging in a specific number for x and working out the result is called evaluating the function.

This is exactly what a model is, in Chapter 1.3's terms: a function, with some of the numbers in its formula left as adjustable parameters instead of fixed constants.

Worked example: evaluating a function by hand

Take the exam-score line from Chapter 1.3, written as a proper function: f(hours) = 6 × hours + 45.

  • f(3) = 6 × 3 + 45 = 63 — a student who studied 3 hours.
  • f(5) = 6 × 5 + 45 = 75 — a student who studied 5 hours.

Same rule both times — only the input changed. Try more values yourself below.

Part II · Chapter 2.2

Vectors — Data as Arrows in Space

Before a model can do anything with a photo, a word, or a movie, it has to first become a list of numbers. That list has a name, and a shape you can actually picture.

🧭 Frame of reference

Giving someone directions — "walk 3 blocks east, then 2 blocks north" — is really just two numbers: 3 and 2. Those two numbers do double duty: they're an instruction (how to move), and they're also a destination (an arrow from where you're standing to exactly one point on the map). Any ordered list of numbers can be read the same way — as an arrow from a starting point to a specific location in space.

A vector is a list of numbers, and also an arrow

A vector is just an ordered list of numbers. With two numbers, you can draw it as an arrow on a flat page — one number for how far along the x-axis, one for how far up the y-axis. Each number in that list describes one feature — one measured property of whatever you're describing. A vector with 3 features is an arrow in 3D space; one with 300 features is still a perfectly valid vector, it's just impossible to draw — the arrow idea is a mental picture that happens to run out at 3 dimensions, not a hard limit on how many numbers a vector can hold.

Worked example: a movie as a vector

Rate a movie's "action-ness" and "romance-ness," each from 0 to 10. Movie A, a loud summer blockbuster, might score (8, 2) — high action, low romance. Movie B, a slow-burn love story, might score (2, 9) — the opposite. Both are 2-dimensional vectors: two numbers, two features, one arrow each. Try building your own below.

Matrices — Tables That Transform Data

A vector is one arrow. Stack several of them together and you get a matrix — which turns out to play two very different roles in ML, often at the same time.

📋 Frame of reference

A matrix is just a spreadsheet: rows and columns of numbers, nothing more exotic than that. But a spreadsheet can also be a machine. Imagine a photocopier with a settings dial — one setting reproduces the page exactly, another stretches it taller, another flips it into a mirror image. Every setting is secretly just a small grid of numbers, and it treats every page you feed it the exact same way, consistently. That's a matrix's other job: not just holding data, but transforming it.

Role one: a matrix as a data table

Stack several vectors on top of each other — one row per movie — and you get a matrix: a table where each row is one example and each column is one feature. The three movies from Chapter 2.2, plus a third one, become a 3-row, 2-column matrix:

Action scoreRomance score
Movie A82
Movie B29
Movie C66

This is exactly what a real dataset looks like inside a computer — rows of examples, columns of features — whether it's 3 movies or 3 million.

Role two: a matrix as a transformation

The other role is the photocopier-dial idea: a matrix can also be something you multiply a vector by to move it — scale it bigger or smaller, flip it, or rotate it — and it treats every vector fed into it the same consistent way. You won't need to compute matrix multiplication by hand in this book; what matters is the intuition, which the demo below shows directly.

Part II · Chapter 2.4

Dot Products & Similarity

How does a recommendation system know Movie A is a better match for you than Movie B? Underneath, it's one small calculation, repeated — the dot product.

🎯 Frame of reference

Imagine comparing your taste in movies to a friend's by going feature by feature: you both rate how much you like action (0–10) and how much you like romance (0–10), multiply your matching scores together, and add the results up. If you both love action and both dislike romance, those products are big and positive, and the total is large — your tastes point in the same direction. If one of you loves what the other hates, the total comes out small. That single running total is exactly what a dot product is.

The calculation

The dot product of two vectors is found by multiplying each pair of matching numbers, then adding everything up. For two 2D vectors (a₁, a₂) and (b₁, b₂), that's a₁×b₁ + a₂×b₂ — one multiplication per matching feature, then a sum. It works the same way no matter how many features the vectors have; you just keep multiplying matching pairs and adding.

Worked example: which movie matches your taste?

Say your taste vector is (7, 3) — you lean toward action over romance. Compare it against the three movies from Chapter 2.2 and 2.3:

  • Movie A (8, 2): 7×8 + 3×2 = 56 + 6 = 62
  • Movie B (2, 9): 7×2 + 3×9 = 14 + 27 = 41
  • Movie C (6, 6): 7×6 + 3×6 = 42 + 18 = 60

Movie A wins, narrowly ahead of Movie C — both lean action, like you do. Movie B, the romance-heavy one, scores noticeably lower. Try changing your taste vector below and watch the ranking shift.

    Probability, the Intuitive Way

    Models rarely say "yes" or "no." They say "73% yes." Understanding what that number actually means — and the two simple rules it follows — is the last piece of math before Part III's real data.

    🌧️ Frame of reference

    "There's a 70% chance of rain tomorrow" is a probability: a number expressing how confident a forecast is, based on past weather patterns. It's not a guarantee — it might not rain at all — but if you saw "70% chance of rain" on a hundred different days, you'd expect it to actually rain on roughly 70 of them. That's the whole idea. Everything else in this chapter is just two rules for working with numbers like that.

    Probability: a number from 0 to 1

    Probability is always a number between 0 and 1 — 0% meaning impossible, 100% meaning certain. There's one rule that always holds: every possible outcome's probabilities add up to exactly 1. If there's a 70% chance of rain, there's a 30% chance of no rain — the two together have to make 100%, since rain-or-no-rain covers every possibility.

    Independent events: multiply

    When two events don't affect each other — independent events — the probability of both happening is found by multiplying their individual probabilities together. Whether it rains tomorrow has nothing to do with how a coin flip lands, so those two are independent.

    Worked example: rain and a coin flip

    There's a 70% chance of rain tomorrow. Independently, a fair coin flip has a 50% chance of landing heads.

    • Probability it does NOT rain: 1 − 0.70 = 0.30 (30%)
    • Probability of rain AND heads: 0.70 × 0.50 = 0.35 (35%)

    Notice it's multiplication, not addition — 70% + 50% would be 120%, which can't be a probability at all.

    Distributions & Uncertainty

    Chapter 2.5 covered the probability of one event. But most things worth measuring have many possible outcomes, not just two — and the shape of how likely each one is has its own name.

    🎲 Frame of reference

    Roll a fair six-sided die a thousand times and chart how often each number comes up — you'd get six roughly equal bars, since each face has the same 1-in-6 chance. Now measure the height of every adult in a city instead. Height isn't limited to six options; it can be almost any value. Chart it anyway, and you get a smooth, bell-shaped hump: most people cluster near the average height, with fewer and fewer people the further you get from it in either direction. Both charts are describing the same kind of thing — how likely each possible outcome is — just for very different situations.

    Distributions: the whole shape, not just one number

    A distribution describes how probability is spread across every possible outcome, not just one. A fair die's distribution is flat — each of the 6 outcomes gets an equal 1/6 share. But distributions don't have to be flat. A "loaded" die weighted heavily toward rolling a 6 has a lopsided distribution — most of the probability piled onto one outcome, with the rest spread thin across the others.

    The normal distribution: a very common shape

    The height example follows a normal distribution — a smooth, symmetric bell shape that shows up constantly, both in nature and in ML. How wide or narrow that bell is describes its spread: a tight, narrow bell means outcomes are predictable and close to average; a wide, flat bell means outcomes vary a lot — more uncertainty. You'll see this exact idea again when neural network weights are initialized (Chapter 6.7) and when diffusion models add and remove noise (Chapter 11.4).

    Part II · Chapter 2.7

    Bayes' Theorem, Explained With a Story

    Here's a question that trips up almost everyone the first time: if a medical test is 99% accurate and you test positive, what's the chance you actually have the condition? The honest answer is rarely 99% — and the reason why is one of the most useful ideas in this entire book.

    🩺 Frame of reference

    Imagine a rare condition that affects just 1% of people. A test for it is 99% accurate — meaning it correctly identifies 99% of people who have it, and correctly clears 99% of people who don't. You take the test. It comes back positive. Most people's gut instinct is "99% chance I have it." The actual answer, it turns out, is closer to a coin flip — about 50%. The test's accuracy isn't the whole story; how rare the condition is matters just as much.

    Why the gut instinct is wrong: count people, not percentages

    Bayes' theorem is easiest to see by imagining 100 real people instead of thinking in abstract percentages. 1% of them — 1 person — actually has the condition. The test correctly flags that 1 person about 99% of the time, so call it 1 true positive. But among the other 99 healthy people, a 99%-accurate test still gets 1% of them wrong — roughly 1 false positive. So out of everyone who tests positive (1 true positive + 1 false positive = 2 people), only half actually have the condition. The test didn't get worse — there just weren't many sick people to begin with, so its small error rate on the huge healthy group produces about as many false alarms as real hits.

    Prior and posterior

    The 1% starting estimate is called the prior — what you believed before any evidence. The ~50% updated estimate, after folding in the positive test result, is the posterior. Bayes' theorem is simply the rule for turning a prior into a posterior correctly, instead of by gut feeling.

    Worked example: a spam filter, revisited

    Suppose only 10% of your incoming mail is truly spam (the prior), and your spam filter is 90% accurate in both directions. Out of 100 emails: 10 are truly spam, 90 are truly legitimate. The filter correctly flags about 9 of the 10 spam emails, and incorrectly flags about 9 of the 90 legitimate ones (90 × 10% ≈ 9). So of the 18 emails flagged as spam (9 real + 9 false), only half are actually spam — even with a filter that sounds quite good on paper.

    ● Interactive demo — try it

    Adjust how rare the condition is and how accurate the test is, out of a sample of 100 people.

    ▸ Python — run in your own environment
    def posterior_probability(rarity, accuracy, population=100):
    diseased = round(population * rarity)
    healthy = population - diseased
    true_positives = round(diseased * accuracy)
    false_positives = round(healthy * (1 - accuracy))
    return true_positives / (true_positives + false_positives)

print(posterior_probability(0.01, 0.99))  # about 0.5 — surprisingly low!
    ✓ What you can now explain to someone else
    • Bayes' theorem updates a prior probability into a posterior probability after new evidence, properly weighing how rare the thing is.
    • A positive test result isn't the same as "you have it" — the test's accuracy has to be weighed against the base rate.
    • Counting people out of a sample (natural frequencies) is often easier than memorizing the formula.
    • The rarer something is, the more a test's false positives can outnumber its true positives.

    Slopes & Change — Intuitive Calculus

    Chapter 1.3's exam-score line had a "steepness" you could turn like a knob. This chapter gives that idea a proper name, and extends it to shapes that aren't straight lines at all.

    🚗 Frame of reference

    Your car's speedometer isn't measuring some separate quantity — it's measuring how fast your position is changing, right at this instant. If you graphed your position over time, that graph would usually be a curve, not a straight line: speeding up here, slowing down there. Your speed at any given moment is exactly the steepness of that curve at that exact point. Calculus is mostly just a name for "the steepness of a curve, measured precisely, at any point you like."

    Slope: rise over run

    Slope measures how much y changes for each unit x changes — "rise over run." It's exactly the m from Chapter 1.3's score = m × hours + b: a bigger m meant a steeper line. For a straight line, the slope is the same everywhere you measure it. That's what makes it a straight line in the first place.

    Curves don't have just one slope

    A curve, unlike a straight line, is steeper in some places than others. So "the slope of the curve" doesn't mean anything on its own — you have to ask for the slope at a specific point. Zoom in closely enough around any single point on a smooth curve, and it starts looking almost exactly like a straight line — that local, zoomed-in slope is called the derivative at that point.

    Worked example: the slope of x²

    Take f(x) = x² — one of the functions from Chapter 2.1's function machine. Its curve is flat near x = 0 and gets steeper the further out you go. It turns out the derivative of x² at any point x is exactly 2x:

    • At x = 1: slope = 2 × 1 = 2 — fairly shallow.
    • At x = 3: slope = 2 × 3 = 6 — much steeper.

    The Chain Rule

    This is the single most important idea in this entire Part — quietly, it's the mathematical trick that makes training a neural network possible at all.

    ⚙️ Frame of reference

    Picture three gears connected in a row. Turn gear A, and it spins gear B twice as fast. Gear B, in turn, spins gear C three times as fast as it spins itself. How much faster does gear C spin compared to gear A? Not 2 + 3 = 5 times. It's 2 × 3 = 6 times as fast — you multiply the ratios along the chain. That's the entire chain rule, no calculus notation required yet.

    Multiply the links in the chain

    The chain rule says: when one quantity depends on another, which depends on another, the overall rate of change from start to end is the product of every link's individual rate of change — exactly like the gears. In derivative notation, if y depends on u, and u depends on x, then dy/dx = dy/du × du/dx. Each d(something)/d(something) is just a slope (Chapter 2.8) — a local rate of change for one link in the chain.

    Worked example: a composed function

    Let u = 2x and y = u² — so y depends on u, and u depends on x. At x = 3: u = 2×3 = 6.

    • du/dx = 2 (constant — u always changes twice as fast as x)
    • dy/du = 2u (from Chapter 2.8's rule for x², applied to u); at u = 6, that's 2×6 = 12
    • Chain rule: dy/dx = dy/du × du/dx = 12 × 2 = 24

    Check it directly: y = (2x)² = 4x², whose slope is 8x (four times the slope of x², since the coefficient 4 comes along for the ride). At x = 3: 8 × 3 = 24. Same answer, two different routes.

    • dy/dx = dy/du × du/dx when y depends on u and u depends on x.

    • A neural network is a chain of layers — backpropagation (Chapter 6.4) is the chain rule applied to that whole stack.

    Variables and functions, now in code

    Chapter 2.1 defined a variable as a name standing in for a value, and a function as a rule that turns an input into an output. In Python, that's not a metaphor — it's literally the syntax:

    • hours = 3 creates a variable named hours holding the value 3.
    • def f(x): return ... defines a function named f.
    • print(...) displays a value so you can see it.

    Two new pieces: strings and loops

    A string is text data, written inside quotes: "spam" or "Movie A", as opposed to a plain number. A loop repeats a block of code once for each item in a list, instead of writing the same line out by hand for every single item.

    Worked example: everything together

    ▸ Python — run in your own environment
    hours_studied = [1, 3, 5, 7]

def predict_score(hours):
    return 6 * hours + 45

for h in hours_studied:
    print(predict_score(h))
